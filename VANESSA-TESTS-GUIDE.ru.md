# Vanessa Automation: правила для агента

> Агентский reference. Читайте этот файл только перед созданием или правкой Vanessa Automation тестов. Не загружайте его для обычных lifecycle-команд.

Цель тестов в ITL-проекте - проверить текущую разрабатываемую фичу. Не пишите общий smoke, полный E2E по системе или длинный пользовательский регресс, если разработчик явно этого не попросил.

## Быстрый алгоритм

1. Определите, какое поведение изменила фича.
2. Выберите самый дешевый тип проверки:
   - `unit-like` - расчет, условие, заполнение, локальная прикладная логика;
   - `integration` - объект, документ, регистр, обмен, движение данных между подсистемами;
   - `UI` - форма, команда, видимое пользовательское поведение.
3. Найдите существующие шаги и локальные `Libraries`/`@exportscenarios`, если они есть в проекте.
4. Напишите 2-3 небольших сценария: основной успешный путь и один значимый граничный или негативный случай. Больше 4 сценариев - только с явным обоснованием.
5. Запустите финальную проверку через `/itl-verify`. Vanessa MCP допустим только для авторинга, поиска шагов, записи и отладки.

## Экономия контекста

- Не читайте большие внешние smoke-наборы перед каждой задачей. Они нужны как справочник паттернов, а не как обязательный контекст.
- Не вставляйте в промпт каталоги известных шагов целиком. Ищите конкретный шаг по смыслу или через Vanessa MCP.
- Не генерируйте длинную UI-последовательность, если поведение можно проверить через данные, объект или регистр.
- Переиспользуйте существующий библиотечный шаг, если он уже выражает бизнес-действие.
- Сценарий держите коротким: подготовка, действие, 1-3 проверяемых результата, очистка при необходимости.
- Не описывайте историю Vanessa, общие возможности фреймворка и чужие наборы тестов в `test-plan.md`.

## Структура feature-файла

Минимальная структура:

```gherkin
#language: ru

@tree
@feature_<change-id>

Функционал: <Короткое имя проверяемой фичи>

Контекст:
	Дано Я запускаю сценарий открытия TestClient или подключаю уже существующий
	И я закрываю все окна клиентского приложения

Сценарий: <Успешный путь>
	* Подготовка
		...

	* Действие
		...

	* Проверка
		...
```

Правила:

- Храните прикладные сценарии в `tests/features`.
- Имена сценариев пишите по проверяемому поведению, а не по внутреннему номеру задачи.
- Используйте `@tree`, если нужны читаемые группы `* Подготовка`, `* Действие`, `* Проверка`.
- Добавляйте `@exportscenarios` только в файлы библиотечных сценариев, которые реально переиспользуются.
- Не добавляйте теги больших smoke/E2E наборов для фичевых проверок.

## Unit-like шаблон

Используйте, когда можно проверить логику без прохода по форме.

```gherkin
#language: ru

@tree
@feature_discount_rule

Функционал: Расчет скидки договора

Контекст:
	Дано Я запускаю сценарий открытия TestClient или подключаю уже существующий

Сценарий: Скидка применяется для договора с действующим условием
	* Подготовка
		И я выполняю код встроенного языка на сервере
		"""bsl
			// Создайте минимальные тестовые данные и сохраните ссылки в переменные контекста.
		"""

	* Действие
		И я выполняю код встроенного языка на сервере
		"""bsl
			// Вызовите проверяемую экспортную процедуру или функцию и сохраните результат в Объект.Результат.
		"""

	* Проверка
		И я выполняю код встроенного языка на сервере
		"""bsl
			Если Объект.Результат <> 100 Тогда
				ВызватьИсключение "Ожидался результат 100";
			КонецЕсли;
		"""
```

## Integration шаблон

Используйте, когда важны запись объекта, проведение документа, движения, регистры или обмен.

```gherkin
#language: ru

@tree
@feature_order_posting

Функционал: Проведение заказа с резервом

Контекст:
	Дано Я запускаю сценарий открытия TestClient или подключаю уже существующий
	И я закрываю все окна клиентского приложения

Сценарий: Проведение заказа создает резерв по складу
	* Подготовка
		И я выполняю код встроенного языка на сервере
		"""bsl
			// Создайте только необходимые справочники, документ и остатки.
		"""

	* Действие
		И я выполняю код встроенного языка на сервере
		"""bsl
			// Проведите документ или вызовите прикладную операцию.
		"""

	* Проверка
		И я выполняю код встроенного языка на сервере
		"""bsl
			// Прочитайте регистр и проверьте результат.
			Если Объект.КоличествоРезерва <> 5 Тогда
				ВызватьИсключение "Ожидался резерв 5";
			КонецЕсли;
		"""
```

## UI шаблон

Используйте только для формы, команды или видимого поведения. Предпочитайте имена элементов формы, а не заголовки и координаты.

```gherkin
#language: ru

@tree
@feature_order_command

Функционал: Команда заполнения заказа

Контекст:
	Дано Я запускаю сценарий открытия TestClient или подключаю уже существующий
	И я закрываю все окна клиентского приложения

Сценарий: Команда Заполнить добавляет строку товара
	* Подготовка
		И я выполняю код встроенного языка на сервере
		"""bsl
			// Создайте минимальные данные и получите навигационную ссылку.
		"""

	* Действие
		И Я открываю навигационную ссылку "$НавигационнаяСсылка$"
		Если появилось предупреждение Тогда
			Тогда я вызываю исключение "Не удалось открыть заказ для проверки команды заполнения"
		Если имя текущей формы "ErrorWindow" Тогда
			Тогда я вызываю исключение "Открылась форма ошибки при открытии заказа"
		И я нажимаю на кнопку с именем 'ФормаЗаполнить'

	* Проверка
		Тогда в таблице "Товары" количество строк "больше" 0
```

## Надежность

- Создавайте минимальные тестовые данные внутри сценария или библиотечного шага. Не полагайтесь на произвольные данные базы, кроме явно согласованных фикстур.
- Имена тестовых объектов делайте уникальными, например с префиксом change-id и текущей датой/UUID.
- Проверяйте не только отсутствие ошибки, но и наблюдаемый результат: значение, запись, движение, строку таблицы, доступность команды.
- После открытия формы проверяйте предупреждения и `ErrorWindow`, если шаг должен продолжаться только при успешном открытии.
- Используйте явные ожидания вроде `я жду закрытия окна ... в течение 20 секунд`. Слепую `пауза` применяйте только когда нет стабильного события, и комментируйте причину.
- Для UI шагов используйте `с именем '<ИмяЭлемента>'`, если имя элемента известно. Текстовые заголовки окон могут меняться.
- Не завершайте и не трогайте чужие `TESTMANAGER`/`TESTCLIENT`; финальный запуск делает helper.

## Библиотеки и самописные шаги

- Выносите в `Libraries` только действие, которое будет переиспользовано минимум двумя сценариями или сильно сокращает шум.
- Библиотечный `.feature` помечайте `@exportscenarios`; прикладной сценарий в `tests/features` вызывает его бизнес-фразой.
- Самописный EPF-шаг добавляйте только если стандартные шаги Vanessa и библиотечный сценарий не покрывают действие надежно.
- Если добавлен самописный шаг, рядом должен быть минимальный `.feature`-пример его вызова.

## Что не делать

- Не писать "дымовой тест всей конфигурации" для проверки одной фичи.
- Не проходить через меню и формы, если проверяемый результат можно получить серверным кодом и assert.
- Не копировать большие сценарии из внешних репозиториев.
- Не создавать больше 4 фичевых проверок без объяснения в `test-plan.md`.
- Не заменять `/itl-verify` запуском через Vanessa MCP, headless EPF или `/deploy-and-test`.
