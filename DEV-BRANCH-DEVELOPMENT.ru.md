# Процесс разработки внутри ветки разработки

Этот документ описывает, как внутри уже созданной ветки разработки использовать AI-агента с правилами `ai_rules_1c` и OpenSpec.

Разработчик выбирает только режим работы:

- **Quick-fix** - для маленьких очевидных исправлений.
- **OpenSpec** - для полноценной разработки изменений.

Роли `ai_rules_1c` разработчик вручную не выбирает. Агент сам выбирает внутреннюю специализацию по задаче.

## Как запускать действия

В Kilo Code жизненный цикл ветки выполняется slash-командами:

```text
/itl-activate-dev-branch-context  Записать базу текущей ветки в .dev.env для команд ai_rules_1c.
/itl-update-dev-branch-base       Обновить базу ветки разработки из текущих файлов.
/itl-run-dev-branch-tests         Запустить тесты Vanessa Automation по базе текущей ветки.
/itl-refresh-dev-branch           Обновить ветку разработки свежим master из хранилища.
/itl-export-dev-branch-result     Выгрузить CF/CFE по текущей ветке без закрытия ветки.
/itl-close-dev-branch             Закрыть ветку разработки, выгрузить финальный CF/CFE и перейти на master.
```

В Codex те же действия можно писать обычным текстом:

```text
Обнови базу текущей ветки разработки.
Запусти тесты текущей ветки разработки через Vanessa Automation.
Обнови текущую ветку разработки из хранилища.
Выгрузи результат по текущей ветке разработки.
Закрой текущую ветку разработки и подготовь финальный результат.
```

Команды жизненного цикла ветки выполняются через `/itl-*` или естественные текстовые команды. Роли остаются внутренней логикой агента.

Если текущая ветка предназначена для разработки расширения, в начале работы задайте имя расширения:

```text
/itl-set-dev-branch-extension <имя-расширения>
```

После создания расширения в копии базы ветки выгрузите его в файлы:

```text
/itl-dump-dev-branch-extension
```

Дальше `/itl-update-dev-branch-base` обновляет расширение в базе ветки из `src/cfe/<имя-расширения>`, а `/itl-export-dev-branch-result` выгружает `CFE`.

Команды жизненного цикла автоматически активируют контекст базы ветки разработки для `ai_rules_1c`. Если разработчик вручную запускает `/update1cbase`, `/loadfrom1cbase` или `/getconfigfiles`, перед этим нужно выполнить `/itl-activate-dev-branch-context`.

`/deploy-and-test` не используется как обычная проверка в ITL-ветке: эта команда повторно загружает все файлы конфигурации. Стандартный путь быстрее и безопаснее: сначала `/itl-update-dev-branch-base`, затем `/itl-run-dev-branch-tests`.

## Как выбрать режим

Используйте **quick-fix**, если выполняются все условия:

- проблема понятна;
- изменение локальное;
- затрагивается один модуль, одна форма или небольшой участок кода;
- не меняется бизнес-процесс;
- не добавляются новые метаданные;
- не меняется публичный интерфейс модулей;
- не требуется согласование архитектуры или подхода.

Примеры quick-fix:

- исправить синтаксическую ошибку;
- поправить условие;
- исправить имя переменной;
- убрать падение в конкретной процедуре;
- добавить простую проверку на `Неопределено`;
- исправить текст сообщения;
- поправить запрос без изменения бизнес-смысла.

Используйте **OpenSpec**, если есть хотя бы один признак:

- новая функция;
- изменение поведения системы;
- изменение метаданных;
- изменение нескольких модулей;
- новая интеграция;
- изменение бизнес-правила;
- изменение прав доступа;
- изменение отчетов, печатных форм или обменов;
- есть риск сломать существующий сценарий;
- нужно сначала согласовать подход.

Если есть сомнение, выбирайте OpenSpec.

## Quick-fix

Quick-fix делается без OpenSpec.

### 1. Поставить задачу

```text
Сделай quick-fix без OpenSpec.

Проблема:
<описание ошибки>

Ожидаемый результат:
<что должно работать>

Ограничения:
- меняй минимально необходимый код;
- не меняй архитектуру;
- не добавляй новые метаданные;
- если выяснится, что задача больше quick-fix, остановись и предложи перейти на OpenSpec.
```

### 2. Реализовать исправление

Агент должен:

- найти место ошибки;
- внести минимальное исправление;
- проверить синтаксис;
- кратко написать, какие файлы изменены;
- описать, чем проверено.

Если во время quick-fix стало понятно, что нужно менять поведение, метаданные или несколько связанных мест, quick-fix прекращается. Дальше задача оформляется через OpenSpec.

### 3. Обновить базу ветки разработки

После изменения файлов обязательно обновите базу ветки разработки.

В Kilo Code:

```text
/itl-update-dev-branch-base
```

В Codex:

```text
Обнови базу текущей ветки разработки.
```

Агент должен обновить копию базы текущей ветки разработки измененными файлами конфигурации.

### 4. Протестировать исправление

Исполняемая проверка делается через Vanessa Automation:

```text
/itl-run-dev-branch-tests
```

Тест не обязан быть UI. Для локальной логики используйте unit-like или интеграционный Vanessa-сценарий; UI-сценарий пишите только когда проверяется пользовательское поведение, форма, команда или видимый результат.

Если тест упал, агент должен сам проанализировать отчет Vanessa, лог 1С и измененный код, исправить ошибку, снова выполнить `/itl-update-dev-branch-base` и перезапустить релевантный сценарий. Лимит - 3 попытки на одну группу ошибок. После 3 неуспешных попыток агент останавливается, фиксирует диагностику и просит решение разработчика.

Только после успешного тестирования можно переходить к выгрузке `CF`, review или закрытию ветки.

## OpenSpec

OpenSpec используем для задач, где нужно описать, согласовать и потом реализовать изменение.

### 1. Создать proposal

```text
Сделай OpenSpec proposal для change-id: <change-id>.

Нужно:
<описание задачи>

Контекст:
<что известно про текущую реализацию>

Ожидаемый результат:
<как должно работать после изменения>
```

На этом шаге агент должен подготовить OpenSpec-артефакты:

- `proposal.md` - что и зачем меняем;
- `design.md` - как технически реализуем, если изменение не тривиальное;
- `tasks.md` - какие конкретно задачи будут выполнены;
- `test-plan.md` - какие Vanessa-сценарии проверяют изменение, какого они типа: unit-like, integration или UI;
- spec deltas - какие требования меняются или добавляются.

Код на этом шаге не меняется.

### 2. Проверить документы OpenSpec

Разработчик проверяет:

- понятно ли, какую проблему решаем;
- соответствует ли proposal исходной задаче;
- достаточно ли конкретный design;
- можно ли по `tasks.md` выполнить реализацию без догадок;
- указаны ли Vanessa-проверки результата и понятно ли, после каких шагов их запускать.

Если план непонятный, попросите уточнить:

```text
Уточни OpenSpec proposal.
Разбей tasks.md на конкретные маленькие шаги с указанием объектов, модулей, процедур и проверок.
Код пока не меняй.
```

### 3. Согласовать реализацию

Код пишется только после того, как разработчик согласовал:

- `proposal.md`;
- `design.md`, если он нужен;
- `tasks.md`;
- изменения требований в spec deltas.

Если решение затрагивает архитектуру, метаданные, интеграции или права, согласование обязательно.

### 4. Реализовать изменение

```text
Реализуй OpenSpec change: <change-id>.
Действуй строго по tasks.md.
После каждого значимого шага отмечай выполненные задачи.
```

Агент реализует задачи из `tasks.md` и отмечает выполненные пункты.

Если изменение большое или состоит из нескольких самостоятельных шагов, агент обязан разбить работу на проверяемые срезы. После каждого значимого среза выполняется цикл: обновить базу ветки, запустить релевантные Vanessa-сценарии, исправить найденные ошибки, затем идти дальше.

### 5. Обновить базу ветки разработки

После реализации маленького изменения или после каждого значимого среза большой задачи обязательно обновите базу ветки разработки.

В Kilo Code:

```text
/itl-update-dev-branch-base
```

В Codex:

```text
Обнови базу текущей ветки разработки.
```

Агент должен обновить копию базы текущей ветки разработки измененными файлами конфигурации. Не запускайте после этого `/deploy-and-test`, потому что он сделает повторную полную загрузку файлов.

### 6. Протестировать сценарий через Vanessa Automation

Vanessa Automation - стандартный тестовый фреймворк ветки разработки. Тесты хранятся в `tests/features`, отчеты - в `build/test-results/vanessa`.

В Kilo Code:

```text
/itl-run-dev-branch-tests
```

В Codex:

```text
Запусти тесты текущей ветки разработки через Vanessa Automation.
```

Минимальный набор проверок:

- синтаксис BSL;
- корректность измененных metadata XML;
- выполнение всех пунктов `tasks.md`;
- отсутствие лишних изменений;
- Vanessa-сценарий, проверяющий измененное бизнес-поведение.

Тип Vanessa-сценария выбирается по сути изменения:

- unit-like - для прикладной логики, расчетов, условий, заполнения данных;
- integration - для взаимодействия объектов, регистров, документов, обменов;
- UI - только для пользовательских форм, команд и видимого поведения.

Если UI-сценарий нужен и база опубликована на Apache, тестируйте через опубликованный web-client URL. Если публикации нет, используйте доступный user-mode сценарий для копии базы ветки или явно зафиксируйте ограничение в отчете.

Если тест упал, агент не должен останавливаться на констатации ошибки. Он анализирует отчет Vanessa, лог 1С, скриншоты и измененный код, исправляет причину, снова выполняет `/itl-update-dev-branch-base` и перезапускает только релевантные сценарии. Лимит - 3 попытки на одну группу ошибок. После 3 неуспешных попыток агент останавливается, фиксирует диагностику и просит решение разработчика.

В `test-report.md` OpenSpec change нужно записать, какие сценарии запускались, какой тип проверки использован, где лежит отчет Vanessa и какие ошибки были исправлены.

Только после успешного тестирования можно переходить к review, выгрузке `CF` или закрытию ветки.

### 7. Code review, если нужно

Code review не запускается автоматически.

Если нужен отдельный review, попросите явно:

```text
Сделай code review изменения <change-id>.
Проверь риски, ошибки, регрессии, недостающие проверки и соответствие OpenSpec.
```

Review должен искать:

- возможные ошибки выполнения;
- регрессии;
- нарушение требований OpenSpec;
- рискованные изменения метаданных;
- недостаточные проверки;
- лишние изменения вне задачи.

### 8. Выгрузить результат или закрыть ветку

Если нужно передать текущую версию изменений в базу, подключенную к хранилищу, но ветку закрывать еще рано:

```text
/itl-export-dev-branch-result
```

Если работа в ветке завершена и протестирована:

```text
/itl-close-dev-branch
```

В Codex можно написать:

```text
Выгрузи результат по текущей ветке разработки.
Закрой текущую ветку разработки и подготовь финальный результат.
```

### 9. Закрыть OpenSpec change

После приемки изменения нужно заархивировать OpenSpec change:

```text
Заархивируй OpenSpec change: <change-id>.
```

Архивация переносит согласованные изменения требований из `openspec/changes/<change-id>/` в актуальные спецификации проекта.

## Короткая памятка

```text
Маленькая локальная ошибка -> quick-fix без OpenSpec.

Новая функция, изменение поведения, метаданные, несколько модулей или риск регрессии -> OpenSpec.

Роли вручную не выбираем. Агент сам выбирает внутреннюю специализацию по задаче.

После изменений всегда обновляем базу ветки разработки: /itl-update-dev-branch-base.

После обновления базы запускаем Vanessa Automation: /itl-run-dev-branch-tests.

Если Vanessa нашла ошибку, агент сам исправляет ее и повторяет проверку. Лимит - 3 попытки.

CF/CFE без закрытия ветки: /itl-export-dev-branch-result.

Закрытие ветки после тестирования: /itl-close-dev-branch.
```
