# Процесс разработки внутри ветки разработки

Этот документ описывает, как внутри уже созданной ветки разработки использовать AI-агента с правилами `ai_rules_1c` и OpenSpec.

В работе есть два основных режима:

- **Quick-fix** — для маленьких очевидных исправлений.
- **OpenSpec** — для полноценной разработки изменений.

Роли из `ai_rules_1c` напрямую обычно не вызываем. Разработчик выбирает режим работы: quick-fix или OpenSpec. Агент сам выбирает нужную роль внутри процесса.

## Как выбрать режим

Используй **quick-fix**, если выполняются все условия:

- проблема понятна;
- изменение локальное;
- затрагивается один модуль, одна форма или небольшой участок кода;
- не меняется бизнес-процесс;
- не добавляются новые метаданные;
- не меняется публичный интерфейс модулей;
- не требуется согласование архитектуры или подхода.

Примеры quick-fix:

- исправить синтаксическую ошибку;
- поправить условие;
- исправить имя переменной;
- убрать падение в конкретной процедуре;
- добавить простую проверку на `Неопределено`;
- исправить текст сообщения;
- поправить запрос без изменения бизнес-смысла.

Используй **OpenSpec**, если есть хотя бы один признак:

- новая функция;
- изменение поведения системы;
- изменение метаданных: реквизиты, документы, справочники, регистры, формы, роли;
- изменение нескольких модулей;
- новая интеграция;
- изменение бизнес-правила;
- изменение прав доступа;
- изменение отчетов, печатных форм или обменов;
- есть риск сломать существующий сценарий;
- нужно сначала согласовать подход.

Если есть сомнение, выбирай OpenSpec.

## Ветка 1. Quick-fix

Quick-fix делается без OpenSpec.

Используй этот режим, когда нужно быстро и минимально исправить конкретную ошибку.

### Шаблон запроса

```text
Сделай quick-fix без OpenSpec.

Проблема:
<описание ошибки>

Ожидаемый результат:
<что должно работать>

Ограничения:
- меняй минимально необходимый код;
- не меняй архитектуру;
- не добавляй новые метаданные;
- если выяснится, что задача больше quick-fix, остановись и предложи перейти на OpenSpec.
```

### Пример запроса

```text
Сделай quick-fix без OpenSpec.

Проблема:
При открытии формы договора падает ошибка, если не заполнен контрагент.

Ожидаемый результат:
Форма должна открываться. Если контрагент не заполнен, поле статуса оплаты остается пустым.

Ограничения:
- меняй минимально необходимый код;
- не меняй метаданные;
- если потребуется менять бизнес-логику, остановись.
```

### Что должен сделать агент

Агент должен:

- найти место ошибки;
- внести минимальное исправление;
- проверить синтаксис;
- кратко написать, какие файлы изменены;
- описать, чем проверено.

Если во время quick-fix стало понятно, что нужно менять поведение, метаданные или несколько связанных мест, quick-fix прекращается. Дальше задача оформляется через OpenSpec.

## Ветка 2. OpenSpec

OpenSpec используем для нормальной разработки изменений.

Через OpenSpec проходят все задачи, где нужно описать, согласовать и потом реализовать изменение.

### Шаг 1. Создать proposal

```text
Сделай OpenSpec proposal для change-id: <change-id>.

Нужно:
<описание задачи>

Контекст:
<что известно про текущую реализацию>

Ожидаемый результат:
<как должно работать после изменения>
```

Пример:

```text
Сделай OpenSpec proposal для change-id: add-contract-payment-status.

Нужно:
Добавить статус оплаты договора.

Контекст:
Сейчас пользователь не видит, оплачен договор или нет.

Ожидаемый результат:
Статус должен отображаться в форме договора и использоваться в отчете по договорам.
```

На этом шаге агент должен подготовить OpenSpec-артефакты:

- `proposal.md` — что и зачем меняем;
- `design.md` — как технически реализуем, если изменение не тривиальное;
- `tasks.md` — какие конкретно задачи будут выполнены;
- spec deltas — какие требования меняются или добавляются.

Код на этом шаге не меняется.

### Шаг 2. Проверить документы OpenSpec

Разработчик проверяет:

- понятно ли, какую проблему решаем;
- соответствует ли proposal исходной задаче;
- достаточно ли конкретный design;
- можно ли по `tasks.md` выполнить реализацию без догадок;
- указаны ли проверки результата.

Если план непонятный, нужно попросить уточнить:

```text
Уточни OpenSpec proposal.
Разбей tasks.md на конкретные маленькие шаги с указанием объектов, модулей, процедур и проверок.
Код пока не меняй.
```

### Шаг 3. Согласовать реализацию

Код пишется только после того, как разработчик согласовал:

- `proposal.md`;
- `design.md`, если он нужен;
- `tasks.md`;
- изменения требований в spec deltas.

Если решение затрагивает архитектуру, метаданные, интеграции или права, согласование обязательно.

### Шаг 4. Реализовать изменение

```text
Реализуй OpenSpec change: <change-id>.
Действуй строго по tasks.md.
После каждого значимого шага отмечай выполненные задачи.
```

Агент реализует задачи из `tasks.md`.

Если основная работа в BSL-коде, агент использует роль разработчика.

Если основная работа в метаданных, агент использует роль менеджера метаданных.

Если задача смешанная, сначала обычно меняются метаданные, затем пишется код.

### Шаг 5. Проверить результат

Минимальный набор проверок:

- синтаксис BSL;
- корректность измененных metadata XML;
- выполнение всех пунктов `tasks.md`;
- отсутствие лишних изменений;
- ручной smoke-test, если задача затрагивает пользовательский сценарий.

Если в проекте есть автоматические тесты или проверки, их нужно запустить.

### Шаг 6. Code review, если нужно

Code review не запускается автоматически.

Если нужен отдельный review, попроси явно:

```text
Сделай code review изменения <change-id>.
Проверь риски, ошибки, регрессии, недостающие проверки и соответствие OpenSpec.
```

Review должен искать:

- возможные ошибки выполнения;
- регрессии;
- нарушение требований OpenSpec;
- рискованные изменения метаданных;
- недостаточные проверки;
- лишние изменения вне задачи.

### Шаг 7. Закрыть изменение

После приемки изменения нужно заархивировать OpenSpec change:

```text
Заархивируй OpenSpec change: <change-id>.
```

Архивация переносит согласованные изменения требований из `openspec/changes/<change-id>/` в актуальные спецификации проекта.

## Как относиться к ролям

Роли из `ai_rules_1c` — это внутренние специализации агента.

В обычном процессе разработчик не должен вручную выбирать роль. Достаточно выбрать ветку: quick-fix или OpenSpec.

Роли нужны как ориентир:

| Роль | Когда нужна |
|---|---|
| `1c-explorer` | Разобраться в существующем коде без изменений. |
| `1c-analytic` | Описать требования, PRD, бизнес-логику. |
| `1c-planner` | Разбить понятную задачу на конкретные шаги. |
| `1c-architect` | Спроектировать сложное изменение, модули, метаданные, потоки данных. |
| `1c-developer` | Писать и менять BSL-код. |
| `1c-metadata-manager` | Создавать и менять метаданные 1С. |
| `1c-refactoring` | Безопасно чистить код и устранять технический долг. |
| `1c-performance-optimizer` | Искать и исправлять проблемы производительности. |
| `1c-error-fixer` | Исправлять конкретные ошибки выполнения или синтаксиса. |
| `1c-tester` | Проверять результат в тестовой базе или через UI-сценарии. |
| `1c-code-reviewer` | Делать отдельный review по явному запросу. |
| `1c-doc-writer` | Писать документацию. |

## Когда выбирать planner или architect

Обычно это делает агент сам.

Если нужно указать явно:

- `1c-planner` — когда задача уже понятна и нужен список конкретных шагов;
- `1c-architect` — когда нужно сначала придумать техническое решение.

Простое правило:

```text
Если ожидаешь список задач — нужен planner.
Если ожидаешь схему решения, границы модулей, поток данных и архитектурные решения — нужен architect.
```

## Когда выбирать developer или metadata-manager

Обычно это делает агент сам.

Если нужно указать явно:

- `1c-developer` — когда меняется BSL-код;
- `1c-metadata-manager` — когда меняется структура конфигурации.

Примеры для `1c-developer`:

- общий модуль;
- модуль объекта;
- модуль формы;
- запрос;
- процедура;
- функция;
- обработчик события.

Примеры для `1c-metadata-manager`:

- справочник;
- документ;
- регистр;
- реквизит;
- табличная часть;
- форма;
- отчет;
- роль;
- макет;
- расширение.

Если задача смешанная, порядок обычно такой:

1. сначала `1c-metadata-manager` меняет структуру;
2. затем `1c-developer` реализует поведение.

## Короткая памятка

```text
Маленькая локальная ошибка → quick-fix без OpenSpec.

Новая функция, изменение поведения, метаданные, несколько модулей или риск регрессии → OpenSpec.

Роли вручную обычно не выбираем.
Агент сам выбирает нужную роль по задаче.

Code review запускаем только отдельным явным запросом.
```
