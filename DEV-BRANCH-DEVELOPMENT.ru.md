# Процесс разработки внутри ветки разработки

> Документ для разработчика. Агентам не нужно читать его как обязательный lifecycle reference: каноническая агентская версия этого процесса находится в `.agents/skills/1c-workflow/references/dev-branch-development.md`. Читайте этот файл только по просьбе пользователя или для подготовки human-facing объяснений.

Этот документ описывает, как внутри уже созданной ветки разработки использовать AI-агента с правилами `ai_rules_1c` и OpenSpec.

Разработчик выбирает только режим работы:

- **Quick-fix** - для маленьких очевидных исправлений.
- **OpenSpec** - для полноценной разработки изменений.

Роли `ai_rules_1c` разработчик вручную не выбирает. Агент сам выбирает внутреннюю специализацию по задаче.

## Как запускать действия

В Kilo Code жизненный цикл ветки выполняется slash-командами:

```text
/itl-vibecoding1c-mcp            Настроить, запустить, обновить или проверить vibecoding1c MCP текущего scope.
/itl-status                       Показать текущую ветку, базу и статус проверки.
/itl-set-dev-branch-extension     Задать имя расширения для текущей extension-ветки.
/itl-dump-dev-branch-extension    Выгрузить файлы расширения из базы текущей ветки.
/itl-update-base                  Обновить базу ветки разработки из текущих файлов.
/itl-verify                       Обновить базу ветки и запустить Vanessa Automation.
/itl-refresh                      Обновить ветку разработки свежим master.
/itl-result                       Выгрузить CF/CFE по текущей ветке без закрытия ветки.
/itl-close                        Закрыть ветку разработки, выгрузить финальный CF/CFE и перейти на master.
```

В Codex те же действия можно писать обычным текстом:

```text
Обнови базу текущей ветки разработки.
Запусти тесты текущей ветки разработки через Vanessa Automation.
Обнови текущую ветку разработки из хранилища.
Выгрузи результат по текущей ветке разработки.
Закрой текущую ветку разработки и подготовь финальный результат.
```

Команды жизненного цикла ветки выполняются через `/itl-*` или естественные текстовые команды. Роли остаются внутренней логикой агента.

`/itl-vibecoding1c-mcp` в worktree текущей `itldev/*` ветки подключает выбранные vibecoding1c MCP endpoints. По умолчанию используются remote LAN endpoints из registry; config-specific remote vibecoding1c MCP требует явного `configId`. Если для `code` или `graph` выбран local branch scope, helper поднимает отдельный локальный vibecoding1c MCP для текущей ветки. vibecoding1c MCP соседних веток в client config не добавляются; Vanessa MCP управляется отдельно через `/itl-vanessa-mcp`.

Команды `/itl-set-dev-branch-extension` и `/itl-dump-dev-branch-extension` являются helper-командами для extension-веток. Они доступны напрямую, но могут не показываться в коротком beginner-меню `/itl`.

Если текущая ветка предназначена для разработки расширения, в начале работы задайте имя расширения:

```text
/itl-set-dev-branch-extension <имя-расширения>
```

После создания расширения в копии базы ветки выгрузите его в файлы:

```text
/itl-dump-dev-branch-extension
```

Дальше `/itl-update-base` обновляет расширение в базе ветки из `src/cfe/<имя-расширения>`, а `/itl-result` выгружает `CFE`.

Команды жизненного цикла автоматически активируют контекст базы ветки разработки для `ai_rules_1c`.

`/deploy-and-test` не используется как обычная проверка в ITL-ветке: эта команда повторно загружает все файлы конфигурации. Стандартный путь быстрее и безопаснее: `/itl-verify`. Если нужно только обновить базу без тестов, используйте `/itl-update-base`.

## Как выбрать режим

Используйте **quick-fix**, если выполняются все условия:

- проблема понятна;
- изменение локальное;
- затрагивается один модуль, одна форма или небольшой участок кода;
- не меняется бизнес-процесс;
- не добавляются новые метаданные;
- не меняется публичный интерфейс модулей;
- не требуется согласование архитектуры или подхода.

Примеры quick-fix:

- исправить синтаксическую ошибку;
- поправить условие;
- исправить имя переменной;
- убрать падение в конкретной процедуре;
- добавить простую проверку на `Неопределено`;
- исправить текст сообщения;
- поправить запрос без изменения бизнес-смысла.

Используйте **OpenSpec**, если есть хотя бы один признак:

- новая функция;
- изменение поведения системы;
- изменение метаданных;
- изменение нескольких модулей;
- новая интеграция;
- изменение бизнес-правила;
- изменение прав доступа;
- изменение отчетов, печатных форм или обменов;
- есть риск сломать существующий сценарий;
- нужно сначала согласовать подход.

Если есть сомнение, выбирайте OpenSpec.

Для одновременной работы по нескольким независимым линиям создавайте отдельные `itldev/*` ветки/worktree. Одна ветка при этом может оставаться долгоживущей и последовательно вести несколько задач, если это осознанная линия работы.

Если разработчику нужно вести несколько независимых линий работы одновременно, для каждой линии создается отдельная `itldev/*` ветка в отдельной worktree-папке. Откройте эту папку отдельным окном Codex/Kilo/IDE и работайте внутри нее.

## Quick-fix

Quick-fix делается без OpenSpec.

### 1. Поставить задачу

```text
Сделай quick-fix без OpenSpec.

Проблема:
<описание ошибки>

Ожидаемый результат:
<что должно работать>

Ограничения:
- меняй минимально необходимый код;
- не меняй архитектуру;
- не добавляй новые метаданные;
- если выяснится, что задача больше quick-fix, остановись и предложи перейти на OpenSpec.
```

### 2. Реализовать исправление

Агент должен:

- найти место ошибки;
- внести минимальное исправление;
- проверить синтаксис;
- кратко написать, какие файлы изменены;
- описать, чем проверено.

Если во время quick-fix стало понятно, что нужно менять поведение, метаданные или несколько связанных мест, quick-fix прекращается. Дальше задача оформляется через OpenSpec.

### 3. Обновить базу ветки разработки

После изменения файлов обязательно обновите базу ветки разработки.

В Kilo Code:

```text
/itl-update-base
```

В Codex:

```text
Обнови базу текущей ветки разработки.
```

Агент должен обновить копию базы текущей ветки разработки измененными файлами конфигурации.

### 4. Протестировать исправление

Исполняемая проверка делается через Vanessa Automation:

```text
/itl-verify
```

Для написания и отладки Vanessa-сценариев агент может запустить branch-local Vanessa MCP через `/itl-vanessa-mcp` из текущей `itldev/*` worktree. MCP не заменяет финальный gate: после отладки нужно выполнить `/itl-verify`, который запускает пакетный `StartFeaturePlayer` в реальном `TESTMANAGER -> TESTCLIENT` контуре, назначает branch-local `VANESSA_TEST_PORT` и сохраняет воспроизводимые JUnit/status/log paths.

Для изменения бизнес-поведения агент обязан создать или обновить небольшой набор Vanessa Automation проверок: минимум 2, обычно 2-3 и не больше 4 без отдельного обоснования. Набор должен включать основной успешный сценарий и минимум один значимый граничный или негативный сценарий.

Тип проверки выбирается по сути изменения: unit-like - для локальной логики, integration - для взаимодействия объектов, регистров, документов или обменов, UI - только для пользовательских форм, команд и видимого поведения. Тест не обязан быть UI.

Если тест упал, агент должен сам проанализировать JUnit/status Vanessa, лог 1С, active 1C process diagnostics и измененный код, исправить ошибку и снова выполнить `/itl-verify`. Если JUnit не создан, это `unknown`, а не pass; сначала проверьте реальный конфликт порта или базы, а не просто наличие чужого `TESTMANAGER`/`TESTCLIENT`. Лимит - 3 попытки на одну группу ошибок. После 3 неуспешных попыток агент останавливается, фиксирует диагностику и просит решение разработчика.

Только после успешного тестирования можно переходить к выгрузке `CF`, review или закрытию ветки.

## OpenSpec

OpenSpec используем для задач, где нужно описать, согласовать и потом реализовать изменение.

### 1. Создать proposal

```text
Сделай OpenSpec proposal для change-id: <change-id>.

Нужно:
<описание задачи>

Контекст:
<что известно про текущую реализацию>

Ожидаемый результат:
<как должно работать после изменения>
```

На этом шаге агент должен подготовить OpenSpec-артефакты:

- `proposal.md` - что и зачем меняем;
- `design.md` - как технически реализуем, если изменение не тривиальное;
- `tasks.md` - какие конкретно задачи будут выполнены;
- `test-plan.md` - какие 2-4 Vanessa-проверки проверяют изменение, какого они типа: unit-like, integration или UI, и какие граничные или негативные случаи они покрывают;
- spec deltas - какие требования меняются или добавляются.

Код на этом шаге не меняется.

### 2. Проверить документы OpenSpec

Разработчик проверяет:

- понятно ли, какую проблему решаем;
- соответствует ли proposal исходной задаче;
- достаточно ли конкретный design;
- можно ли по `tasks.md` выполнить реализацию без догадок;
- указаны ли Vanessa-проверки результата и понятно ли, после каких шагов их запускать.

Если план непонятный, попросите уточнить:

```text
Уточни OpenSpec proposal.
Разбей tasks.md на конкретные маленькие шаги с указанием объектов, модулей, процедур и проверок.
Код пока не меняй.
```

### 3. Согласовать реализацию

Код пишется только после того, как разработчик согласовал:

- `proposal.md`;
- `design.md`, если он нужен;
- `tasks.md`;
- изменения требований в spec deltas.

Если решение затрагивает архитектуру, метаданные, интеграции или права, согласование обязательно.

### 4. Реализовать изменение

```text
Реализуй OpenSpec change: <change-id>.
Действуй строго по tasks.md.
После каждого значимого шага отмечай выполненные задачи.
```

Агент реализует задачи из `tasks.md` и отмечает выполненные пункты.

Если изменение большое или состоит из нескольких самостоятельных шагов, агент обязан разбить работу на проверяемые срезы. После каждого значимого среза выполняется цикл: обновить базу ветки, запустить релевантные Vanessa-сценарии, исправить найденные ошибки, затем идти дальше.

### 5. Обновить базу ветки разработки

После реализации маленького изменения или после каждого значимого среза большой задачи обязательно обновите базу ветки разработки.

В Kilo Code:

```text
/itl-update-base
```

В Codex:

```text
Обнови базу текущей ветки разработки.
```

Агент должен обновить копию базы текущей ветки разработки измененными файлами конфигурации. Не запускайте после этого `/deploy-and-test`, потому что он сделает повторную полную загрузку файлов.

### 6. Протестировать сценарий через Vanessa Automation

Vanessa Automation - стандартный тестовый фреймворк ветки разработки. Тесты хранятся в `tests/features`, отчеты - в `build/test-results/vanessa`.

Vanessa MCP - опциональная branch-local поддержка для авторинга и отладки. Запускайте отдельный MCP для каждой ветки разработки; порт сохраняется в state ветки, поэтому параллельные worktree не конфликтуют. Финальная проверка не идет через MCP: `/itl-verify` запускает Vanessa как `TESTMANAGER -> TESTCLIENT` с отдельным test port текущей ветки.

Remote `code`/`graph` MCP и project-scope local MCP могут быть построены по снимку конфигурации, который не включает текущие изменения ветки. Перед использованием данных MCP смотрите `/itl-status`: freshness `remote-shared`, `stale`, `unknown` или `indexing` означает, что MCP не является точным индексом текущей branch head.

В Kilo Code:

```text
/itl-verify
```

В Codex:

```text
Запусти тесты текущей ветки разработки через Vanessa Automation.
```

Минимальный набор проверок:

- синтаксис BSL;
- корректность измененных metadata XML;
- выполнение всех пунктов `tasks.md`;
- отсутствие лишних изменений;
- 2-4 Vanessa-проверки измененного бизнес-поведения: основной успешный сценарий и минимум один значимый граничный или негативный сценарий.

Тип Vanessa-сценария выбирается по сути изменения:

- unit-like - для прикладной логики, расчетов, условий, заполнения данных;
- integration - для взаимодействия объектов, регистров, документов, обменов;
- UI - только для пользовательских форм, команд и видимого поведения.

Если изменение затрагивает взаимодействие объектов, регистров, документов или обменов, включите интеграционную проверку в этот набор.

Если UI-сценарий нужен и база опубликована на Apache, тестируйте через опубликованный web-client URL. Если публикации нет, используйте доступный user-mode сценарий для копии базы ветки или явно зафиксируйте ограничение в отчете.

Если тест упал, агент не должен останавливаться на констатации ошибки. Он анализирует JUnit/status Vanessa, лог 1С, скриншоты, active 1C process diagnostics и измененный код, исправляет причину и снова выполняет `/itl-verify` для релевантного сценария. Чужие процессы worktree по умолчанию не являются причиной ждать; ищите реальный конфликт порта или базы. Не завершайте процессы чужих worktree; helper может остановить только собственный зависший `TESTCLIENT`, совпадающий с текущей базой/портом. Лимит - 3 попытки на одну группу ошибок. После 3 неуспешных попыток агент останавливается, фиксирует диагностику и просит решение разработчика.

В `test-report.md` OpenSpec change нужно записать, какие сценарии запускались, какой тип проверки использован, где лежит отчет Vanessa и какие ошибки были исправлены.

Только после успешного тестирования можно переходить к review, выгрузке `CF` или закрытию ветки.

### 7. Code review, если нужно

Code review не запускается автоматически.

Если нужен отдельный review, попросите явно:

```text
Сделай code review изменения <change-id>.
Проверь риски, ошибки, регрессии, недостающие проверки и соответствие OpenSpec.
```

Review должен искать:

- возможные ошибки выполнения;
- регрессии;
- нарушение требований OpenSpec;
- рискованные изменения метаданных;
- недостаточные проверки;
- лишние изменения вне задачи.

### 8. Выгрузить результат или закрыть ветку

Если нужно передать текущую версию изменений в базу, подключенную к хранилищу, но ветку закрывать еще рано:

```text
/itl-result
```

Если работа в ветке завершена и протестирована:

```text
/itl-close
```

В Codex можно написать:

```text
Выгрузи результат по текущей ветке разработки.
Закрой текущую ветку разработки и подготовь финальный результат.
```

Если нет свежей успешной проверки Vanessa, агент должен предупредить об этом. Получить результат или закрыть ветку без такой проверки можно только после явного подтверждения разработчика.

Идеальный промышленный контур должен запрещать `result/close` без fresh passed Vanessa, code review и заполненного `test-report.md`. В текущем упрощенном процессе это остается предупреждением с явным подтверждением, а рядом с `CF`/`CFE` создается `<artifact>.manifest.json` со статусом проверки, SHA256, commit-ами, логами и признаком unverified override.

### 9. Закрыть OpenSpec change

После приемки изменения нужно заархивировать OpenSpec change:

```text
Заархивируй OpenSpec change: <change-id>.
```

Архивация переносит согласованные изменения требований из `openspec/changes/<change-id>/` в актуальные спецификации проекта.

## Короткая памятка

```text
Маленькая локальная ошибка -> quick-fix без OpenSpec.

Новая функция, изменение поведения, метаданные, несколько модулей или риск регрессии -> OpenSpec.

Роли вручную не выбираем. Агент сам выбирает внутреннюю специализацию по задаче.

После изменений можно обновить базу ветки разработки: /itl-update-base.

Стандартная проверка: /itl-verify. Она запускает Vanessa через `TESTMANAGER -> TESTCLIENT`, а не через MCP.

Если Vanessa нашла ошибку, агент сам исправляет ее и повторяет /itl-verify. Лимит - 3 попытки.

CF/CFE без закрытия ветки: /itl-result.

Закрытие ветки: /itl-close.

Рядом с каждым выгруженным CF/CFE создается JSON manifest результата.
```
