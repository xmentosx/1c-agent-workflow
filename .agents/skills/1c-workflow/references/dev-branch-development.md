# Процесс разработки внутри ветки разработки

Этот документ описывает, как внутри уже созданной ветки разработки использовать AI-агента с правилами `ai_rules_1c` и OpenSpec.

Разработчик выбирает только режим работы:

- **Quick-fix** - для маленьких очевидных исправлений.
- **OpenSpec** - для полноценной разработки изменений.

Роли `ai_rules_1c` разработчик вручную не выбирает. Агент сам выбирает внутреннюю специализацию по задаче.

## Как запускать действия

В Kilo Code жизненный цикл ветки выполняется slash-командами:

```text
vibecoding1c-mcp helper action            Настроить, запустить, обновить или проверить vibecoding1c MCP текущего scope.
/itl-status                       Показать текущую ветку, базу и статус проверки.
/itl-check                        Обновить базу ветки и запустить Vanessa Automation.
update-base helper action                  Обновить базу ветки без тестов.
verify helper action                       Совместимый alias для /itl-check.
/itl-refresh                      Обновить ветку разработки свежим master.
/itl-result                       Выгрузить CF/CFE по текущей ветке.
```

В Kilo Code OpenSpec-процесс запускается отдельными slash-командами:

```text
/opsx-propose                     Обычный первый шаг: подготовить proposal, design/tasks/test-plan/spec deltas; код не менять.
/opsx-apply                       Реализовать согласованный OpenSpec change по tasks.md.
/opsx-archive                     После приемки заархивировать OpenSpec change.
/opsx-explore                     Опционально (optional): исследовать задачу или код до proposal, если контекста недостаточно.
```

В Codex те же действия можно писать обычным текстом:

```text
Обнови базу текущей ветки разработки.
Запусти тесты текущей ветки разработки через Vanessa Automation.
Обнови текущую ветку разработки из хранилища.
Выгрузи результат по текущей ветке разработки.
Закрой текущую ветку разработки и подготовь финальный результат.
```

Команды жизненного цикла ветки выполняются через `/itl-*` или естественные текстовые команды. Роли остаются внутренней логикой агента.

`vibecoding1c-mcp helper action` в worktree текущей `itldev/*` ветки подключает выбранные vibecoding1c MCP endpoints. Если для `code` или `graph` выбран local branch scope, helper поднимает отдельный vibecoding1c MCP для текущей ветки. vibecoding1c MCP соседних веток в client config не добавляются; Vanessa MCP управляется отдельно через `vanessa-mcp helper action`.

Если текущая ветка предназначена для разработки расширения, в начале работы задайте имя расширения:

```text
set-dev-branch-extension helper action <имя-расширения>
```

После создания расширения в копии базы ветки выгрузите его в файлы:

```text
dump-dev-branch-extension helper action
```

Дальше `/itl-check` обновляет расширение в базе ветки из `src/cfe/<имя-расширения>` и запускает Vanessa Automation, а `/itl-result` выгружает `CFE`. Если нужно только обновить базу без тестов, используйте `update-base helper action`.

Команды жизненного цикла автоматически активируют контекст базы ветки разработки для `ai_rules_1c`.

`/deploy-and-test` не используется как обычная проверка в ITL-ветке: эта команда повторно загружает все файлы конфигурации. Стандартный путь быстрее и безопаснее: `/itl-check`. Если нужно только обновить базу без тестов, используйте `update-base helper action`. `verify helper action` остается совместимым alias.

After a real file load, `/itl-check`, `update-base helper action`, `verify helper action`, `/itl-refresh`, `/itl-result`, and advanced `close-dev-branch` automatically launch the branch infobase in Enterprise user mode through bundled `ДляАвтоматическогоОбновленияИБ.epf`. This applies update handlers and answers the legal-copy prompt non-interactively. No-op updates do not launch Enterprise.

## Как выбрать режим

Используйте **quick-fix**, если выполняются все условия:

- проблема понятна;
- изменение локальное;
- затрагивается один модуль, одна форма или небольшой участок кода;
- не меняется бизнес-процесс;
- не добавляются новые метаданные;
- не меняется публичный интерфейс модулей;
- не требуется согласование архитектуры или подхода.

Примеры quick-fix:

- исправить синтаксическую ошибку;
- поправить условие;
- исправить имя переменной;
- убрать падение в конкретной процедуре;
- добавить простую проверку на `Неопределено`;
- исправить текст сообщения;
- поправить запрос без изменения бизнес-смысла.

Используйте **OpenSpec**, если есть хотя бы один признак:

- новая функция;
- изменение поведения системы;
- изменение метаданных;
- изменение нескольких модулей;
- новая интеграция;
- изменение бизнес-правила;
- изменение прав доступа;
- изменение отчетов, печатных форм или обменов;
- есть риск сломать существующий сценарий;
- нужно сначала согласовать подход.

Если есть сомнение, выбирайте OpenSpec.

## Quick-fix

Quick-fix делается без OpenSpec.

### 1. Поставить задачу

```text
Сделай quick-fix без OpenSpec.

Проблема:
<описание ошибки>

Ожидаемый результат:
<что должно работать>

Ограничения:
- меняй минимально необходимый код;
- не меняй архитектуру;
- не добавляй новые метаданные;
- если выяснится, что задача больше quick-fix, остановись и предложи перейти на OpenSpec.
```

### 2. Реализовать исправление

Агент должен:

- найти место ошибки;
- внести минимальное исправление;
- проверить синтаксис;
- кратко написать, какие файлы изменены;
- описать, чем проверено.

Если во время quick-fix стало понятно, что нужно менять поведение, метаданные или несколько связанных мест, quick-fix прекращается. Дальше задача оформляется через OpenSpec.

### 3. Обновить базу ветки разработки

После изменения файлов обязательно обновите базу ветки разработки.

В Kilo Code:

```text
/itl-check
```

В Codex:

```text
Обнови базу текущей ветки разработки.
```

Агент должен обновить копию базы текущей ветки разработки измененными файлами конфигурации.

### 4. Протестировать исправление

Исполняемая проверка делается через Vanessa Automation:

```text
/itl-check
```

Для изменения бизнес-поведения агент обязан создать или обновить небольшой набор Vanessa Automation проверок: минимум 2, обычно 2-3 и не больше 4 без отдельного обоснования. Набор должен включать основной успешный сценарий и минимум один значимый граничный или негативный сценарий.

Тип проверки выбирается по сути изменения: unit-like - для локальной логики, integration - для взаимодействия объектов, регистров, документов или обменов, UI - только для пользовательских форм, команд и видимого поведения. Тест не обязан быть UI.

Если тест упал, агент должен сам проанализировать отчет Vanessa, лог 1С и измененный код, исправить ошибку и снова выполнить `/itl-check`. Лимит - 3 попытки на одну группу ошибок. После 3 неуспешных попыток агент останавливается, фиксирует диагностику и просит решение разработчика.

Только после успешного тестирования можно переходить к выгрузке `CF` или review.

## OpenSpec

OpenSpec используем для задач, где нужно описать, согласовать и потом реализовать изменение.

Обычно начинайте сразу с `/opsx-propose`. `/opsx-explore` нужен только когда перед proposal не хватает понимания текущего кода, границ задачи или возможного подхода.

### 0. Исследовать контекст, если нужно

```text
/opsx-explore
```

Этот шаг опционален. Агент исследует код, текущую реализацию и границы задачи, но не создает proposal и не меняет код.

### 1. Создать proposal

```text
/opsx-propose

Сделай OpenSpec proposal для change-id: <change-id>.

Нужно:
<описание задачи>

Контекст:
<что известно про текущую реализацию>

Ожидаемый результат:
<как должно работать после изменения>
```

На этом шаге агент должен подготовить OpenSpec-артефакты:

- `proposal.md` - что и зачем меняем;
- `design.md` - как технически реализуем, если изменение не тривиальное;
- `tasks.md` - какие конкретно задачи будут выполнены;
- `test-plan.md` - какие 2-4 Vanessa-проверки проверяют изменение, какого они типа: unit-like, integration или UI, и какие граничные или негативные случаи они покрывают;
- spec deltas - какие требования меняются или добавляются.

Код на этом шаге не меняется.

### 2. Проверить документы OpenSpec

Разработчик проверяет:

- понятно ли, какую проблему решаем;
- соответствует ли proposal исходной задаче;
- достаточно ли конкретный design;
- можно ли по `tasks.md` выполнить реализацию без догадок;
- указаны ли Vanessa-проверки результата и понятно ли, после каких шагов их запускать.

Если план непонятный, попросите уточнить:

```text
Уточни OpenSpec proposal.
Разбей tasks.md на конкретные маленькие шаги с указанием объектов, модулей, процедур и проверок.
Код пока не меняй.
```

### 3. Согласовать реализацию

Код пишется только после того, как разработчик согласовал:

- `proposal.md`;
- `design.md`, если он нужен;
- `tasks.md`;
- изменения требований в spec deltas.

Если решение затрагивает архитектуру, метаданные, интеграции или права, согласование обязательно.

### 4. Реализовать изменение

```text
/opsx-apply

Реализуй OpenSpec change: <change-id>.
Действуй строго по tasks.md.
После каждого значимого шага отмечай выполненные задачи.
```

Агент реализует задачи из `tasks.md` и отмечает выполненные пункты.

Если изменение большое или состоит из нескольких самостоятельных шагов, агент обязан разбить работу на проверяемые срезы. После каждого значимого среза выполняется цикл: выполнить `/itl-check`, исправить найденные ошибки, затем идти дальше.

### 5. Обновить базу ветки разработки

После реализации маленького изменения или после каждого значимого среза большой задачи обязательно обновите базу ветки разработки.

В Kilo Code:

```text
/itl-check
```

В Codex:

```text
Обнови базу текущей ветки разработки.
```

Агент должен выполнить `/itl-check`: helper обновит копию базы текущей ветки разработки измененными файлами конфигурации и запустит Vanessa Automation. Не запускайте после этого `/deploy-and-test`, потому что он сделает повторную полную загрузку файлов.

### 6. Протестировать сценарий через Vanessa Automation

Vanessa Automation - стандартный тестовый фреймворк ветки разработки. Тесты хранятся в `tests/features`, отчеты - в `build/test-results/vanessa`.

В Kilo Code:

```text
/itl-check
```

В Codex:

```text
Запусти тесты текущей ветки разработки через Vanessa Automation.
```

Минимальный набор проверок:

- синтаксис BSL;
- корректность измененных metadata XML;
- выполнение всех пунктов `tasks.md`;
- отсутствие лишних изменений;
- 2-4 Vanessa-проверки измененного бизнес-поведения: основной успешный сценарий и минимум один значимый граничный или негативный сценарий.

Тип Vanessa-сценария выбирается по сути изменения:

- unit-like - для прикладной логики, расчетов, условий, заполнения данных;
- integration - для взаимодействия объектов, регистров, документов, обменов;
- UI - только для пользовательских форм, команд и видимого поведения.

Если изменение затрагивает взаимодействие объектов, регистров, документов или обменов, включите интеграционную проверку в этот набор.

Если UI-сценарий нужен и база опубликована для web-клиента, тестируйте через опубликованный URL. Если публикации нет, используйте доступный user-mode сценарий для копии базы ветки или явно зафиксируйте ограничение в отчете.

Если тест упал, агент не должен останавливаться на констатации ошибки. Он анализирует отчет Vanessa, лог 1С, скриншоты и измененный код, исправляет причину и снова выполняет `/itl-check` для релевантного сценария. Лимит - 3 попытки на одну группу ошибок. После 3 неуспешных попыток агент останавливается, фиксирует диагностику и просит решение разработчика.

В `test-report.md` OpenSpec change нужно записать, какие сценарии запускались, какой тип проверки использован, где лежит отчет Vanessa и какие ошибки были исправлены.

Только после успешного тестирования можно переходить к review или выгрузке `CF`.

### 7. Code review, если нужно

Code review не запускается автоматически.

Если нужен отдельный review, попросите явно:

```text
Сделай code review изменения <change-id>.
Проверь риски, ошибки, регрессии, недостающие проверки и соответствие OpenSpec.
```

Review должен искать:

- возможные ошибки выполнения;
- регрессии;
- нарушение требований OpenSpec;
- рискованные изменения метаданных;
- недостаточные проверки;
- лишние изменения вне задачи.

### 8. Выгрузить результат

Если нужно передать текущую версию изменений в базу, подключенную к хранилищу:

```text
/itl-result
```

В Codex можно написать:

```text
Выгрузи результат по текущей ветке разработки.
Подготовь финальный результат по текущей ветке разработки.
```

Если нет свежей успешной проверки Vanessa, helper применяет `VERIFICATION_POLICY`. По умолчанию `warn`: агент должен предупредить об этом, а получить результат без такой проверки можно только после явного подтверждения разработчика. В режиме `block` helper запрещает result до fresh passed `/itl-check` или `verify helper action`.

### 9. Закрыть OpenSpec change

После приемки изменения нужно заархивировать OpenSpec change:

```text
/opsx-archive

Заархивируй OpenSpec change: <change-id>.
```

Архивация переносит согласованные изменения требований из `openspec/changes/<change-id>/` в актуальные спецификации проекта.

## Короткая памятка

```text
Маленькая локальная ошибка -> quick-fix без OpenSpec.

Новая функция, изменение поведения, метаданные, несколько модулей или риск регрессии -> OpenSpec.

Роли вручную не выбираем. Агент сам выбирает внутреннюю специализацию по задаче.

После изменений запускайте единый цикл проверки: /itl-check.

Если нужно только обновить базу без тестов: update-base helper action.

Стандартная проверка идет через `/itl-check`; `verify helper action` остается совместимым alias.

Если Vanessa нашла ошибку, агент сам исправляет ее и повторяет /itl-check. Лимит - 3 попытки.

CF/CFE текущей ветки: /itl-result.
```
