# 1C Agent Workflow

Пакет правил и команд для доработки 1С-конфигураций с помощью агентов в Codex и Kilo Code.

Он помогает стандартизировать процесс:

- инициализация проекта из базы 1С, подключенной к хранилищу конфигурации;
- выгрузка конфигурации в Git;
- разработка каждой фичи в отдельной ветке;
- работа в отдельной копии базы, отвязанной от хранилища;
- обновление фичи свежей версией из хранилища;
- подготовка промежуточного или финального `CF`;
- переключение между `master` и ветками подпроектов.

## Полная инструкция

Основной документ для разработчика:

```text
DEVELOPER-GUIDE.ru.md
```

В нем описан полный рабочий процесс: первый запуск, старт фичи, загрузка изменений в базу подпроекта, обновление из хранилища, выгрузка `CF`, завершение фичи и переключение веток.

Отдельная инструкция по самой разработке внутри ветки фичи:

```text
FEATURE-DEVELOPMENT.ru.md
```

В ней описано, когда использовать quick-fix, когда оформлять изменение через OpenSpec, как согласовывать proposal/tasks и как запускать review.

## Что входит в пакет

- `.agents/skills/1c-workflow` - общий Agent Skill для Codex и Kilo Code.
- `.kilo/commands` - slash-команды для Kilo Code.
- `AGENT-INSTALL.md` - bootstrap-инструкция для агента.
- `DEVELOPER-GUIDE.ru.md` - инструкция для разработчика.
- `FEATURE-DEVELOPMENT.ru.md` - процесс разработки фичи внутри ветки.
- `templates/project.json` - шаблон настроек проекта без секретов.
- `templates/tools.json` - список проверяемого софта и подсказки по установке.
- `templates/dev.env.example` - пример локальных настроек и секретов.
- PowerShell-helper для Git, 1С Designer, копий баз, Apache-публикации и выгрузки `CF`.

## Первый запуск

Откройте папку будущего проекта в Codex или Kilo Code и попросите агента:

```text
Инициализируй 1С-проект по файлу https://raw.githubusercontent.com/xmentosx/1c-agent-workflow/master/AGENT-INSTALL.md
```

Используйте именно полную ссылку на файл. Так агенту не нужно угадывать ветку репозитория и искать `AGENT-INSTALL.md` самостоятельно.

Текущая папка агента считается папкой проекта. Агент покажет ее абсолютный путь, попросит подтвердить продолжение, проверит установленный софт, создаст локальный Git-репозиторий, обновит исходную базу из хранилища 1С, выгрузит конфигурацию в `src/cf` ветки `master` и установит проектные правила для текущего агента.

После успешной инициализации `src/cf/ConfigDumpInfo.xml` должен быть в `HEAD` ветки `master`. Каталог `src/cf` хранится в Git; большие `CF/DT` и локальные базы игнорируются отдельно.

Если агенту нужны параметры базы и хранилища, он должен спросить их одной группой отдельных коротких вопросов после выбора типа базы. Если среда не умеет показывать группу вопросов, агент должен задавать эти вопросы по одному. Не вводите 6 или 7 строк в одно текстовое поле: в Codex/Kilo Code Enter может сразу отправить первую строку. Отвечайте одним значением на каждый вопрос, без имен переменных и без формата `KEY=value`.

При инициализации агент также спросит, нужно ли публиковать базы фич на локальном Apache для тестирования через веб-клиент. Ответ хранится только в локальном `.dev.env` как `WEB_PUBLISH_BY_DEFAULT=true` или `false`. Если выбрать “нет”, Apache не трогается. Если выбрать “да”, агент сам запускает определение Apache: берет `webinst.exe` рядом с выбранным `1cv8.exe`, ищет установленный Apache/httpd и выводит найденные значения для `.dev.env`. Если Apache не найден, агент должен спросить разрешение на автоустановку и после согласия выполнить `install-apache`: helper скачает официальный zip Apache Lounge, распакует его в `C:\Apache24`, настроит порт, зарегистрирует службу `Apache24` и повторит определение. Параметры Apache вручную в обычном сценарии вводить не нужно.

Для файловой базы анкета должна содержать 6 отдельных вопросов: путь к каталогу базы, пользователь базы, пароль базы или `нет`/`-`, адрес хранилища, пользователь хранилища, пароль хранилища или `нет`/`-`.

Для серверной базы анкета должна содержать 7 отдельных вопросов: сервер 1С, имя базы, пользователь базы, пароль базы или `нет`/`-`, адрес хранилища, пользователь хранилища, пароль хранилища или `нет`/`-`.

Для платформы 1С агент сначала ищет установленные версии в существующих каталогах `C:\Program Files\1cv8` и `C:\Program Files (x86)\1cv8` и предлагает выбрать одну из них. Если одного из каталогов нет, это не ошибка. Полный путь к `1cv8.exe` вводится вручную только если нужная версия не найдена.

В строках пароля точные значения `нет` и `-` означают пустой пароль. Агент должен сохранить пустое значение, а не текст `нет` или `-`.

Файлы `.dev.env` и `.agent-1c/*.json` должны записываться в UTF-8. Это важно для путей и пользователей с кириллицей.

Копии баз фич по умолчанию создаются внутри проекта в `.agent-1c/infobases/features`. Каталог `.agent-1c/infobases/` добавляется в `.gitignore`, поэтому базы не попадают в Git.

## Основные команды Kilo Code

```text
/1c                 Показать меню действий.
/1c-init            Инициализировать проект.
/1c-start <name>    Начать новую фичу.
/1c-load            Загрузить измененные файлы фичи в базу подпроекта.
/1c-refresh         Обновить текущую фичу свежим master из хранилища.
/1c-cf              Сделать промежуточный CF по текущей версии фичи.
/1c-sync            Обновить только master из хранилища.
/1c-finish          Завершить текущую фичу, сделать финальный CF и вернуться на master.
/1c-features        Показать фичи в разработке и текущую фичу.
/1c-master          Переключиться на master.
/1c-feature <name>  Переключиться на ветку фичи.
```

## Использование в Codex

В Codex можно выбрать skill:

```text
/skills -> 1C Workflow
```

Или писать обычным текстом:

```text
Начни 1С-фичу "Скидки по заказам"
Обнови фичу свежей версией из хранилища
Сделай CF по текущей версии фичи
Заверши фичу и подготовь финальный CF
Покажи фичи в разработке
Переключись на master
```

## Важные правила

- Не коммитьте `.dev.env`, пароли, `*.cf`, `*.dt` и логи.
- Не загружайте изменения фичи напрямую в исходную базу, подключенную к хранилищу.
- Все изменения фичи загружаются только в базу подпроекта.
- Перед переключением веток Git-дерево должно быть чистым.
- После полного завершения фичи агент должен переключить проект обратно на `master`.
- Установка софта не выполняется автоматически: агент только проверяет наличие и предлагает команды установки.

## Распространение

Разместите этот пакет в публичном репозитории. Разработчики будут ссылаться на публичный `AGENT-INSTALL.md` при создании нового 1С-проекта.
