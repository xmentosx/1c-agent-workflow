# Разработка изменения

Документ описывает, как разработчику поставить задачу агенту и контролировать результат внутри созданной `itldev/*` ветки.

## Перед началом

Откройте worktree нужной ветки и проверьте контекст:

```text
/itl-status
```

Опишите требуемое поведение, ограничения и критерии приемки обычным текстом. Если задача относится к конкретному тикету или продуктовой документации, укажите ссылку или идентификатор.

## Выбор процесса

Агент механически классифицирует изменение.

### Quick-fix

Подходит для одного понятного локального изменения в одном модуле либо изолированного небольшого изменения метаданных без архитектурного или транзакционного риска. Лимит BSL-строк задает `QUICKFIX_MAX_LINES`, по умолчанию `40`.

Обычный запрос:

```text
Исправь <наблюдаемая проблема>. Ожидаемое поведение: <результат>.
```

Quick-fix не требует OpenSpec, но не отменяет тестирование. Нужен как минимум один focused Vanessa regression scenario; второй добавляется только для отдельной значимой границы или негативного случая.

### OpenSpec

Используется для новой функциональности, изменения поведения, нескольких модулей, связанных метаданных, неоднозначных требований или заметного риска регрессии.

OpenSpec доступен в двух режимах. `native` означает, что pinned upstream установил для активного клиента managed command или SKILL. `natural` означает, что общий `openspec/` workspace и правила исправны, но native bundle для клиента отсутствует. Артефакты, preflight и completion gate в обоих режимах одинаковы. Узнать текущий режим и точный способ запуска можно через `/itl`.

Для natural-режима используйте точные запросы:

```text
Исследуй задачу в режиме OpenSpec, не создавая proposal и не меняя код
```

```text
Подготовь OpenSpec proposal для <изменение>; создай proposal, design, tasks, test-plan и spec deltas; код не меняй
```

```text
Реализуй согласованный OpenSpec change <change-id> по tasks.md и test-plan.md
```

```text
Заархивируй принятый OpenSpec change <change-id> и синхронизируй specs
```

В native-режиме `/itl` вместо этих запросов показывает реально установленные upstream commands или skills. Не переносите синтаксис одного клиента на другой и не считайте `/opsx*` универсальным.

До реализации проверьте proposal, design, спецификации, `tasks.md` и `test-plan.md`. В них должны быть зафиксированы границы изменения, критерии приемки, `Context Sources` и проверяемые сценарии. Apply в обоих режимах владеет штатной проверкой реализованных срезов и финальным fresh passed `/itl-check`. Не запускайте `/itl-check` повторно сразу после успешного apply, если после него не было новых правок; повтор нужен только после более позднего изменения или stale/failed evidence.

Отсутствие внешнего `openspec` executable не блокирует natural-режим. Workflow не выполняет скрытый `npm install` и не запускает `openspec update`; эти операции могут создать неуправляемые файлы поверх pinned upstream snapshot.

Если контекст пока неясен в native-клиенте с соответствующей командой:

```text
/opsx-explore
```

Для подготовки согласуемого изменения в таком native-клиенте:

```text
/opsx-propose <описание>
```

После согласования в таком native-клиенте:

```text
/opsx-apply <change-id>
```

Эти `/opsx-*` примеры относятся только к клиентам, где `/itl` показывает именно такой native-синтаксис.

## Реализация и проверяемые срезы

Для большой задачи не нужен полный прогон после каждой строки `tasks.md`. Работу делят на проверяемые срезы:

- срез с наблюдаемым поведением получает focused Vanessa scenario;
- подготовительная задача без самостоятельного поведения отмечается как pending verification;
- ближайший следующий проверяемый срез покрывает накопленное изменение;
- после последней задачи выполняется полный набор сценариев change.

Для OpenSpec обычно достаточно 2–3 Vanessa-сценариев: основной успешный путь и значимые границы или негативные случаи. Четвертый сценарий требует краткого обоснования.

Тип теста выбирается по поведению:

- unit-like — расчеты, заполнение и локальная прикладная логика;
- integration — документы, регистры, объекты и обмены;
- UI — формы, команды и видимое пользователю состояние.

## Штатная проверка

После quick-fix, ручных изменений вне apply или правок, сделанных после последней проверки, выполните:

```text
/itl-check
```

Команда загружает измененные файлы в копию базы и запускает разрешенную Vanessa Automation и проверку журнала регистрации. Fresh passed результат относится к точному содержимому конфигурации, расширения и тестов; следующая правка делает его stale.

Не запускайте отдельное обновление базы перед `/itl-check`. Не заменяйте executable evidence данными MCP или ручным открытием формы.

Если агент мог пропустить релевантный сценарий либо должен диагностировать падение, исправить реализацию и повторить проверку, используйте:

```text
/itl-verify-fix
```

Сначала переиспользуется существующее покрытие. Новый сценарий создается только при реальном пробеле. После трех неуспешных попыток по одной группе ошибок агент останавливается и показывает диагностику.

## Режимы проверки

`/litemode` и `/itl-litemode` управляют разными слоями. Первый меняет глубину upstream статических проверок, второй — выполнение ITL Vanessa/event-log компонентов. Их точные значения и defaults приведены в [справочнике режимов](MODES-AND-SETTINGS.ru.md).

Если ITL-компонент явно отключен, результат считается partial, а не verified. При `VERIFICATION_POLICY=warn` partial result требует подтверждения; при `block` он запрещен.

## Code review

Отдельный review не запускается автоматически. Если он нужен, попросите явно:

```text
Сделай code review изменения <change-id>. Проверь ошибки, регрессии, риски, недостающие тесты и соответствие OpenSpec.
```

После исправлений review предыдущий `/itl-check` становится stale; выполните проверку снова.

## Результат и завершение

Когда требуется передать текущую версию изменения:

```text
/itl-result
```

Проверьте путь к `CF`/`CFE`, manifest, SHA256 и статус verification. Импорт результата в исходную базу выполняется вручную.

После приемки OpenSpec change заархивируйте его способом, который показывает `/itl`. Для native-клиента с таким синтаксисом:

```text
/opsx-archive <change-id>
```

Для natural-режима:

```text
Заархивируй принятый OpenSpec change <change-id> и синхронизируй specs
```

Архивация переносит согласованные изменения спецификаций в актуальные документы проекта. Она выполняется после приемки, а не вместо проверки или выгрузки результата.

## Критерий готовности

Изменение можно назвать готовым, когда:

- реализованы согласованные требования;
- добавлено или обновлено релевантное Vanessa-покрытие;
- после последних правок получен fresh passed `/itl-check`;
- указан путь к отчету Vanessa;
- для OpenSpec обновлены tasks/test report и выполнена нужная архивация.

Если тест нельзя реализовать или проверку нельзя довести до pass, результат оформляется как блокер или явно подтвержденный partial result, но не как verified/done.
