# Работа с проектом 1C Agent Workflow

Этот документ описывает повседневную работу с уже установленным workflow: где находится актуальная конфигурация, как создавать ветки, проверять изменения, обновляться и получать результат.

Примеры ниже используют slash-команды клиентов, которые их поддерживают. Всего поддерживаются десять single-client вариантов: Codex, Kilo Code, Claude Code, Cursor, OpenCode, Kimi Code, Qwen Code, Command Code, Cline и Pi. Kimi вызывает ITL routines как `/skill:itl-*`, Cline — через skills, Pi — через prompts и требует trust проекта для pinned MCP extension. Точный интерфейс активного клиента показывает `/itl`.

OpenSpec имеет отдельные режимы: `native`, когда upstream managed bundle действительно установлен, и `natural`, когда доступны общий workspace и правила без native shortcuts. Для natural режима фазы запускаются обычными запросами из [руководства по разработке изменения](FEATURE-DEVELOPMENT.ru.md). `/opsx*` не универсален; отсутствие внешнего OpenSpec CLI не блокирует natural flow и не запускает скрытую установку.

## Модель проекта

- `master` хранит актуальную выгрузку исходной конфигурации в Git. Разработка прикладного кода и метаданных непосредственно в `master` запрещена.
- `itldev/<name>` — изолированная ветка разработки со своей worktree-папкой и копией информационной базы.
- Агент применяет изменения только к копии базы текущей `itldev/*` ветки. Исходная база изменяется разработчиком вручную после получения `CF` или `CFE`.

Основная папка проекта остается на `master`. После создания ветки откройте выданный helper путь к worktree отдельным окном AI-клиента или IDE.

## После инициализации

В `master` должны быть закоммичены настройки проекта и исходники конфигурации. Обычно используются:

| Путь | Назначение | Git |
|---|---|---|
| `src/cf` | Файлы основной конфигурации | tracked |
| `src/cfe/<Имя>` | Файлы расширений | tracked |
| `tests/features` | Vanessa-сценарии | tracked |
| `.agent-1c/project.json` | Несекретные настройки проекта | tracked |
| `.agent-1c/dependency-lock.json` | Зафиксированные зависимости и provenance | tracked |
| `.dev.env` | Локальные параметры и секреты | ignored |
| `.agent-1c/infobases/dev-branches` | Копии баз веток | ignored |
| `.agent-1c/dev-branches` | Runtime-состояние веток | ignored |
| `build/result` | Выгруженные `CF`/`CFE` и manifest | ignored |
| `build/test-results/vanessa` | Отчеты Vanessa Automation | ignored |

Полный перечень локальных параметров приведен в [справочнике `.dev.env`](DEV-ENV-REFERENCE.ru.md).

## Создание ветки разработки

Из `master` попросите агента создать ветку конфигурации или расширения. В клиентах со slash-командами доступны:

```text
/itl-new-config-branch <name>
/itl-new-extension-branch <name>
```

Helper проверит чистоту Git, создаст `itldev/<name>` в соседней worktree-папке, подготовит копию базы и сообщит новый путь. Для просмотра активных веток и путей используйте:

```text
/itl-status
```

### Ветка основной конфигурации

В configuration-ветке агент изменяет файлы под настроенным `exportPath` — обычно `src/cf`. Одна ветка может последовательно содержать несколько связанных задач, но свежесть проверки всегда оценивается по текущему содержимому.

### Ветка расширения

При создании extension-ветки агент заранее спросит, создать пустое расширение или загрузить существующий `CFE`, а также имя расширения и путь к CFE при необходимости. Helper создаст worktree и копию базы, затем сам выполнит отдельную транзакционную фазу инициализации в новой ветке. Если параметры пока неизвестны, ветка сохранит статус ожидания и агент соберёт их при первом входе; запускать PowerShell вручную не нужно.

Одна extension-ветка обслуживает ровно одно изменяемое расширение. Для другого `CFE` создайте отдельную ветку.

## Работа с изменением

Поставьте задачу обычным текстом. Агент сам определит, подходит ли quick-fix или нужен OpenSpec. Подробный процесс описан в [руководстве по разработке изменения](FEATURE-DEVELOPMENT.ru.md).

После изменений стандартная проверка выполняется одной командой:

```text
/itl-check
```

Она обновляет копию базы измененными файлами, применяет обработчики обновления в пользовательском режиме и запускает разрешенные компоненты проверки. Отдельное обновление базы перед `/itl-check` не требуется.

Если нужно только загрузить файлы без тестового прогона, попросите агента выполнить `update-base helper action`. Если предыдущая реализация могла пропустить тест или проверка упала и требуется цикл исправления, используйте `/itl-verify-fix`.

## Обновление свежим master

Когда исходная база или `master` изменились, из worktree текущей ветки выполните:

```text
/itl-refresh
```

Helper синхронизирует источник, обновит `master`, сольет его в текущую ветку и обновит копию базы. При Git-конфликте операция останавливается: конфликт нужно разрешить явно, после чего повторить штатный цикл.

## Получение результата

Для передачи текущего состояния в исходную базу выполните:

```text
/itl-result
```

Helper выгрузит `CF` для configuration-ветки или `CFE` для extension-ветки в `build/result`. Рядом создается manifest с SHA256, commit-ами, статусом проверки и путями к отчетам. Загрузка полученного файла в исходную базу выполняется вручную.

По умолчанию `VERIFICATION_POLICY=warn`: при отсутствии свежей успешной проверки потребуется явное подтверждение непроверенного результата. В режиме `block` выгрузка запрещена до fresh passed `/itl-check`.

## Обслуживание workflow

Из чистой `master`:

- `/itl-update-workflow` обновляет managed-файлы workflow и правила;
- `/itl-switch-client <client>` переключает активный AI-клиент;
- `/itl-status` показывает provenance, состояние инструментов и активные worktree.

Обновление не переносится автоматически в существующие `itldev/*` ветки. Для каждой активной ветки используйте `/itl-refresh` после review и commit изменений в `master`.

## Правила безопасности

- Не коммитьте `.dev.env`, пароли, локальные базы, скачанные инструменты, логи и `build/`.
- Не загружайте файлы ветки разработки напрямую в исходную базу через helper.
- Перед lifecycle-операциями сохраняйте чистое tracked Git-дерево.
- Не завершайте процессы 1С, принадлежащие другой worktree.
- ROCTUP MCP и Vanessa UI MCP запускаются по требованию и не заменяют `/itl-check`.
- При ошибке 1С или Git сохраните диагностику и путь к логу; не продолжайте lifecycle вручную в обход helper.
