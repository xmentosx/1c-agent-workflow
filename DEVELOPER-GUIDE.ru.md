# Инструкция разработчика

Эта инструкция описывает стандартный процесс доработки 1С-конфигурации с помощью агента.

## 1. Основная идея

В проекте есть две сущности:

- `master` - локальная Git-ветка со свежей выгрузкой конфигурации из исходной базы. Исходная база может быть подключена к хранилищу 1С или обновляться вручную.
- `itldev/<name>` - ветка разработки для параллельной работы. У нее есть собственная копия базы 1С, отвязанная от хранилища только если исходная база была подключена к нему.

Ветка разработки может использоваться не только под одну бизнес-фичу. Допустим сценарий, когда в одной ветке последовательно выполняется несколько задач, а изменения периодически передаются в исходную базу через выгруженный `CF`.

Агент никогда не загружает изменения из ветки разработки напрямую в исходную базу. Для переноса изменений агент готовит `CF`, а загрузка в исходную базу выполняется вручную.

## 2. Инициализация проекта

Откройте пустую или существующую папку проекта в Codex или Kilo Code и напишите:

```text
Инициализируй 1С-проект по файлу https://raw.githubusercontent.com/xmentosx/1c-agent-workflow/master/AGENT-INSTALL.md
```

Текущая папка будет папкой проекта. Агент должен показать абсолютный путь и спросить, продолжать ли инициализацию именно в ней.

Во время инициализации агент спросит:

- версию платформы 1С, если ее нельзя выбрать из найденных установок;
- тип исходной базы: файловая или серверная;
- параметры исходной базы и, если она подключена к хранилищу, параметры хранилища;
- нужно ли публиковать базы веток разработки на Apache для веб-клиента.

Агент отдельно спросит, подключена ли исходная база к хранилищу конфигурации. Если не подключена, вопросы про адрес, пользователя и пароль хранилища не задаются.

Для файловой базы, подключенной к хранилищу, параметры базы и хранилища задаются группой отдельных вопросов:

1. Путь к каталогу базы.
2. Пользователь базы.
3. Пароль базы или `нет`/`-`.
4. Адрес хранилища.
5. Пользователь хранилища.
6. Пароль хранилища или `нет`/`-`.

Для файловой базы без хранилища задаются только 3 вопроса: путь к каталогу базы, пользователь базы, пароль базы или `нет`/`-`.

Для серверной базы, подключенной к хранилищу:

1. Сервер 1С.
2. Имя базы.
3. Пользователь базы.
4. Пароль базы или `нет`/`-`.
5. Адрес хранилища.
6. Пользователь хранилища.
7. Пароль хранилища или `нет`/`-`.

Значения `нет` и `-` в поле пароля означают пустой пароль.

Для серверной базы без хранилища задаются только 4 вопроса: сервер 1С, имя базы, пользователь базы, пароль базы или `нет`/`-`.

После инициализации в `master` должен быть закоммичен каталог `src/cf` с файлом `src/cf/ConfigDumpInfo.xml`.

## 3. Где что хранится

- `src/cf` - выгрузка конфигурации в файлы, хранится в Git.
- `src/cfe` - выгрузки расширений в файлы, хранятся в Git.
- `.agent-1c/project.json` - настройки проекта без секретов.
- `.dev.env` - локальные настройки и секреты, не коммитится.
- `.agent-1c/dev-branches/*.json` - состояние веток разработки.
- `.agent-1c/infobases/dev-branches` - копии файловых баз веток разработки, не коммитятся.
- `build/result` - выгруженные `CF`/`CFE`, не коммитятся.
- `logs/1c` - логи запусков 1С, не коммитятся.

## 4. Создание ветки разработки

В Kilo Code:

```text
/itl-new-dev-branch СкидкиПоЗаказам
```

Или обычным текстом:

```text
Создай ветку разработки "Скидки по заказам"
```

Агент должен:

1. Проверить, что Git-дерево чистое.
2. Переключиться на `master`.
3. Создать ветку `itldev/<name>`.
4. Скопировать исходную базу в `.agent-1c/infobases/dev-branches/<name>`.
5. Отвязать копию базы от хранилища, если исходная база была подключена к хранилищу.
6. Добавить копию базы в список баз 1С в папку `/ITL/<имя-папки-проекта>`.
7. При включенной Apache-публикации опубликовать базу.
8. Сохранить состояние ветки в `.agent-1c/dev-branches/<name>.json`.

Все дальнейшие изменения загружаются только в базу этой ветки разработки.

### Ветка разработки расширения

Если в ветке нужно разрабатывать расширение, создайте отдельную extension-ветку:

```text
/itl-new-extension-dev-branch МоеРасширение
```

При создании ветки имя расширения не указывается и само расширение в базе не создается. В начале разработки задайте имя расширения:

```text
/itl-set-dev-branch-extension МоеРасширение
```

Затем создайте расширение в копии базы этой ветки через Конфигуратор. После создания выгрузите его в файлы:

```text
/itl-dump-dev-branch-extension
```

Файлы расширения появятся в `src/cfe/<имя-расширения>`. Дальше изменения в файлах расширения обновляются в базе ветки обычной командой `/itl-update-dev-branch-base`, а итоговый `CFE` выгружается через `/itl-export-dev-branch-result`.

## 5. Работа внутри ветки

Процесс разработки внутри ветки описан отдельно:

```text
DEV-BRANCH-DEVELOPMENT.ru.md
```

Короткое правило:

- маленькое локальное исправление можно делать как quick-fix;
- более широкие, рискованные или неоднозначные изменения оформляются через OpenSpec.

После изменения файлов конфигурации обновите базу ветки:

```text
/itl-update-dev-branch-base
```

Или:

```text
Обнови базу текущей ветки разработки
```

Агент обновляет базу ветки только измененными файлами из `src/cf`, формируя `listFile` и передавая его в 1С через `/LoadConfigFromFiles ... -listFile ... -Format Hierarchical`.

## 6. Обновление ветки свежим master

Если в хранилище или в исходной базе появились изменения от других разработчиков:

```text
/itl-refresh-dev-branch
```

Агент должен:

1. Проверить чистоту Git-дерева.
2. Если исходная база подключена к хранилищу, обновить ее из хранилища. Если не подключена, пропустить этот шаг: перед командой разработчик должен сам перенести свежие изменения в исходную базу.
3. Выгрузить свежий `src/cf` в `master`.
4. Закоммитить изменения `master`, если они есть.
5. Вернуться в текущую ветку разработки.
6. Смержить `master` в ветку разработки.
7. Обновить базу ветки измененными файлами.

Если возникнут конфликты, агент должен остановиться и помочь их разрешить.

## 7. Выгрузка CF без закрытия ветки

Когда нужно передать текущую версию изменений в исходную базу:

```text
/itl-export-dev-branch-result
```

Агент:

1. Проверит чистоту Git-дерева.
2. Убедится, что работает с текущей веткой разработки.
3. Обновит базу ветки измененными файлами.
4. Выгрузит `CF` или `CFE` в `build/result`.
5. Сообщит путь к файлу.

Эта команда не обновляет `master` и не закрывает ветку.

## 8. Список и переключение веток

Показать активные ветки разработки:

```text
/itl-list-dev-branches
```

Переключиться на `master`:

```text
/itl-switch-master
```

Переключиться на ветку разработки:

```text
/itl-switch-dev-branch СкидкиПоЗаказам
```

Переключение ветки является Git-действием. Оно не обновляет базу 1С автоматически. Если после переключения нужно обновить базу ветки, выполните `/itl-update-dev-branch-base`.

## 9. Закрытие ветки разработки

Когда работа в ветке завершена и протестирована:

```text
/itl-close-dev-branch
```

Агент должен:

1. Проверить чистоту Git-дерева.
2. Обновить `master` из хранилища или из текущего состояния исходной базы.
3. Смержить `master` в ветку разработки.
4. Обновить базу ветки итоговыми изменениями.
5. Выгрузить финальный `CF` или `CFE` в `build/result`.
6. Записать `closedAt` в состояние ветки.
7. Сообщить путь к `CF`, commit ветки, commit `master` и путь к последнему логу 1С.
8. Переключиться на `master`.

## 10. Полезные команды

```text
/itl                              Показать меню действий.
/itl-init-project                 Инициализировать проект.
/itl-new-dev-branch <name>        Создать ветку разработки конфигурации.
/itl-new-extension-dev-branch <name> Создать ветку разработки расширения.
/itl-set-dev-branch-extension <name> Задать имя расширения для текущей ветки.
/itl-dump-dev-branch-extension    Выгрузить расширение текущей ветки в src/cfe.
/itl-update-dev-branch-base       Обновить базу текущей ветки разработки из файлов ветки.
/itl-refresh-dev-branch           Обновить текущую ветку свежим master.
/itl-export-dev-branch-result     Выгрузить CF/CFE по текущей ветке без закрытия.
/itl-sync-master                  Обновить только master.
/itl-close-dev-branch             Закрыть текущую ветку и вернуться на master.
/itl-list-dev-branches            Показать активные ветки разработки.
/itl-switch-master                Переключиться на master.
/itl-switch-dev-branch <name>     Переключиться на ветку разработки.
```

Для ежедневных операций можно использовать быстрые команды. Они сразу запускают PowerShell-helper и обычно тратят меньше времени и токенов агента:

```text
/itlx                             Показать быстрое меню.
/itlx-init-project                Быстро попробовать инициализацию; при сложной анкете использовать /itl-init-project.
/itlx-new-dev-branch <name>       Создать ветку разработки конфигурации.
/itlx-new-extension-dev-branch <name> Создать ветку разработки расширения.
/itlx-set-dev-branch-extension <name> Задать имя расширения для текущей ветки.
/itlx-dump-dev-branch-extension   Выгрузить расширение текущей ветки в src/cfe.
/itlx-update-dev-branch-base      Обновить базу текущей ветки разработки из файлов ветки.
/itlx-refresh-dev-branch          Обновить текущую ветку свежим master.
/itlx-export-dev-branch-result    Выгрузить CF/CFE по текущей ветке без закрытия.
/itlx-sync-master                 Обновить только master.
/itlx-close-dev-branch            Закрыть текущую ветку и вернуться на master.
/itlx-list-dev-branches           Показать активные ветки разработки.
/itlx-switch-master               Переключиться на master.
/itlx-switch-dev-branch <name>    Переключиться на ветку разработки.
```

## 11. Правила безопасности

- Не коммитьте `.dev.env`, пароли, `*.cf`, `*.dt`, логи и локальные базы.
- Не загружайте изменения ветки разработки напрямую в исходную базу.
- Перед переключением веток, обновлением `master` и выгрузкой `CF` Git-дерево должно быть чистым.
- Если 1С или Git вернули ошибку, агент должен остановиться и показать путь к логу.
