# Инструкция разработчика: 1C Agent Workflow

Эта инструкция описывает, как разработчику дорабатывать 1С-конфигурацию с помощью агента, Git, рабочей копии базы и хранилища конфигурации 1С.

## 1. Идея процесса

В проекте всегда есть две разные роли базы:

- Исходная база подключена к хранилищу конфигурации 1С. Агент использует ее только для получения свежей версии конфигурации из хранилища и выгрузки файлов в `master`.
- База подпроекта создается как копия исходной базы, отвязывается от хранилища и используется для разработки и тестирования конкретной фичи.

Git используется как рабочая история изменений:

- `master` хранит свежую выгрузку конфигурации из исходной базы, подключенной к хранилищу.
- Каждая фича разрабатывается в отдельной ветке `feature/<имя-фичи>`.
- Изменения фичи загружаются только в базу подпроекта.
- Для переноса изменений в базу, подключенную к хранилищу, агент готовит `CF`.

Важно: агент не загружает изменения фичи напрямую в исходную базу, подключенную к хранилищу. Финальный перенос в хранилище выполняется вручную через подготовленный `CF`.

## 2. Первый запуск проекта

Откройте папку будущего проекта в Codex или Kilo Code и попросите агента:

```text
Инициализируй 1С-проект по файлу https://raw.githubusercontent.com/xmentosx/1c-agent-workflow/master/AGENT-INSTALL.md
```

Используйте именно полную ссылку на файл. Так агенту не нужно угадывать ветку репозитория и искать `AGENT-INSTALL.md` самостоятельно.

Текущая папка агента считается папкой проекта. Перед инициализацией агент должен показать ее абсолютный путь и спросить, продолжать ли инициализацию именно в этой папке.

Агент должен спросить недостающие параметры:

- Значения базы и хранилища вводятся одной анкетой после выбора типа базы. Отвечайте значениями по строкам в указанном порядке, без имени переменной и без формата `KEY=value`.
- версия платформы 1С. Агент сначала ищет установленные версии в существующих каталогах `C:\Program Files\1cv8` и `C:\Program Files (x86)\1cv8` и предлагает выбрать одну из них. Если одного из каталогов нет, это нормально. Полный путь к `1cv8.exe` вводится вручную только если нужная версия не найдена;
- тип исходной базы: файловая или серверная;
- для файловой базы агент должен задать одну анкету на 6 строк: путь к каталогу базы, пользователь базы, пароль базы или `нет`/`-`, адрес хранилища, пользователь хранилища, пароль хранилища или `нет`/`-`;
- для серверной базы агент должен задать одну анкету на 7 строк: сервер 1С, имя базы, пользователь базы, пароль базы или `нет`/`-`, адрес хранилища, пользователь хранилища, пароль хранилища или `нет`/`-`. Агент сформирует строку подключения как `Srvr="server";Ref="base";`;
- в строках пароля точные значения `нет` и `-` означают пустой пароль. Не вводите текст "без пароля";
- нужно ли использовать Apache-публикацию для веб-клиента.

Выбор Apache-публикации является локальной настройкой разработчика и хранится в `.dev.env` как `WEB_PUBLISH_BY_DEFAULT=true` или `false`. Если вы отвечаете “нет”, агент не должен спрашивать пути Apache. Если отвечаете “да”, агент одной анкетой спрашивает путь к `webinst.exe`, тип Apache, каталог публикаций, базовый URL и опциональный путь к `httpd.conf`. Когда `webinst.exe` найден рядом с выбранным `1cv8.exe`, агент предлагает этот путь по умолчанию.

После этого агент:

1. Проверит необходимый софт.
2. Создаст локальный Git-репозиторий, если его еще нет.
3. Переключится на локальную ветку `master`.
4. Обновит исходную базу из хранилища конфигурации 1С.
5. Выгрузит конфигурацию в файлы в `src/cf` и закоммитит ее в `master`.
6. Установит правила `ai_rules_1c`.
7. Установит workflow-команды для текущего агента.
8. Создаст служебные файлы проекта.

Первая выгрузка в пустой `src/cf` выполняется полностью. Последующие выгрузки идут инкрементально через механизм 1С `-update -force`, если в `src/cf` есть `ConfigDumpInfo.xml`.

После инициализации в `master` обязательно должен быть закоммичен файл `src/cf/ConfigDumpInfo.xml`. Каталог `src/cf` является частью проекта в Git, даже если в `.gitignore` есть широкие правила для других служебных или build-каталогов.

Если какого-то софта не хватает, агент покажет команды установки или настройки. Он не должен устанавливать софт без вашего явного подтверждения.

## 3. Где хранятся настройки

После инициализации в проекте появятся файлы:

- `.agent-1c/project.json` - общие настройки проекта без секретов.
- `.agent-1c/tools.json` - список проверяемого софта и подсказки по установке.
- `.agent-1c/features/*.json` - состояние фич: ветка, путь к базе подпроекта, публикация.
- `.dev.env` - локальные пути, пользователи и пароли. Этот файл не коммитится.
- `.agents/skills/1c-workflow/` - общий workflow для Codex и Kilo Code.
- `.kilo/commands/` - slash-команды для Kilo Code.

Копии баз фич по умолчанию создаются внутри проекта в `.agent-1c/infobases/features`. Каталог `.agent-1c/infobases/` добавляется в `.gitignore`, поэтому базы не коммитятся.

В `.dev.env` пустой пароль записывается как пустое значение: `IB_PASSWORD=` или `REPOSITORY_PASSWORD=`. В анкете можно написать `нет` или `-`, но в `.dev.env` агент должен сохранить пустую строку, а не этот текст.

Файлы `.dev.env` и `.agent-1c/*.json` должны быть в UTF-8. Тогда русские имена пользователей и пути к базам, например `D:\Git\PM5 КОРП 4`, передаются в 1С без кракозябр.

Не коммитьте `.dev.env`, `*.cf`, `*.dt`, логи и локальные артефакты.

## 4. Как посмотреть доступные действия

В Kilo Code наберите:

```text
/1c
```

В Codex выберите skill:

```text
/skills -> 1C Workflow
```

Или пишите обычным текстом:

```text
Что можно сделать в этом 1С-проекте?
```

## 5. Начало разработки фичи

Когда нужно начать новую доработку, попросите агента:

```text
Начни 1С-фичу "Скидки по заказам"
```

В Kilo Code можно использовать:

```text
/1c-start Скидки по заказам
```

Агент должен:

1. Проверить, что Git-дерево чистое.
2. Переключиться на `master`.
3. Создать ветку `feature/<имя-фичи>`.
4. Скопировать исходную базу в `.agent-1c/infobases/features/<имя-фичи>`, если не задан другой каталог.
5. Отвязать копию базы от хранилища конфигурации.
6. При необходимости опубликовать копию на Apache.
7. Сохранить состояние фичи в `.agent-1c/features/`.

После этого все изменения по фиче выполняются в ветке фичи и загружаются только в базу подпроекта.

## 6. Обычная работа над фичей

В рабочем цикле разработчик и агент меняют файлы конфигурации в ветке фичи. Сам процесс разработки описан отдельно в `FEATURE-DEVELOPMENT.ru.md`: там указано, когда делать quick-fix без OpenSpec, а когда оформлять изменение через OpenSpec proposal/tasks.

Когда нужно загрузить измененные файлы в базу подпроекта:

```text
Загрузи текущую фичу в базу подпроекта
```

В Kilo Code:

```text
/1c-load
```

Агент загрузит в базу подпроекта только измененные файлы конфигурации из `src/cf` и выполнит обновление конфигурации базы данных. Для этого он формирует `listFile` в `logs/1c` и передает его в 1С через `/LoadConfigFromFiles ... -listFile ... -Format Hierarchical`.

После успешной проверки изменений коммитьте их в ветку фичи обычным Git-коммитом.

## 7. Обновление фичи свежей версией из хранилища

Если во время работы в хранилище 1С появились изменения от других разработчиков, обновите фичу:

```text
Обнови текущую фичу свежей версией из хранилища
```

В Kilo Code:

```text
/1c-refresh
```

Агент должен:

1. Проверить чистоту Git-дерева.
2. Переключиться на `master`.
3. Обновить исходную базу из хранилища 1С.
4. Выгрузить свежую конфигурацию в файлы.
5. Закоммитить изменения в `master`, если они есть.
6. Переключиться обратно на ветку фичи.
7. Смержить `master` в ветку фичи.
8. Загрузить результат в базу подпроекта.

Если возникнут конфликты, агент должен остановиться и помочь их разрешить. После разрешения конфликтов нужно снова загрузить конфигурацию в базу подпроекта.

## 8. Промежуточный CF по фиче

Иногда нужно передать текущую версию изменений в базу, подключенную к хранилищу, еще до полного завершения фичи. Для этого попросите:

```text
Сделай CF по текущей версии фичи
```

В Kilo Code:

```text
/1c-cf
```

Агент должен:

1. Проверить чистоту Git-дерева.
2. Переключиться на ветку фичи, если вы сейчас не на ней.
3. Загрузить измененные файлы ветки в базу подпроекта.
4. Выгрузить `CF` из базы подпроекта.
5. Сообщить путь к файлу `CF`.

Эта команда не обновляет `master` из хранилища и не делает merge. Если нужен свежий `master`, сначала выполните обновление фичи из хранилища.

## 9. Переключение между master и фичами

Чтобы посмотреть фичи в разработке и текущую активную фичу:

```text
Покажи фичи в разработке
```

В Kilo Code:

```text
/1c-features
```

Чтобы перейти на основную ветку:

```text
Переключись на master
```

В Kilo Code:

```text
/1c-master
```

Чтобы перейти на ветку фичи:

```text
Переключись на фичу "Скидки по заказам"
```

В Kilo Code:

```text
/1c-feature Скидки по заказам
```

Переключение веток не загружает файлы в 1С автоматически. Если после переключения нужно обновить базу подпроекта, выполните загрузку фичи отдельно.

## 10. Полное завершение фичи

Когда разработка и тестирование закончены:

```text
Заверши текущую фичу и подготовь CF
```

В Kilo Code:

```text
/1c-finish
```

Агент должен:

1. Убедиться, что фича протестирована.
2. Проверить чистоту Git-дерева.
3. Обновить `master` из хранилища 1С.
4. Смержить `master` в ветку фичи.
5. Загрузить итоговую конфигурацию в базу подпроекта.
6. Выгрузить финальный `CF`.
7. Сообщить путь к `CF`, commit ветки, commit `master`, путь к последнему логу 1С.
8. Переключиться обратно на `master`.

Финальный `CF` используется для ручной загрузки изменений в исходную базу, подключенную к хранилищу конфигурации.

## 11. Когда использовать какую команду

```text
/1c                 Показать меню действий.
/1c-init            Инициализировать проект.
/1c-start <name>    Начать новую фичу.
/1c-load            Загрузить измененные файлы фичи в базу подпроекта.
/1c-refresh         Обновить текущую фичу свежим master из хранилища.
/1c-cf              Сделать промежуточный CF по текущей версии фичи.
/1c-sync            Обновить только master из хранилища.
/1c-finish          Завершить текущую фичу, сделать финальный CF и вернуться на master.
/1c-features        Показать фичи в разработке и текущую фичу.
/1c-master          Переключиться на master.
/1c-feature <name>  Переключиться на ветку фичи.
```

В Codex можно писать те же действия обычным текстом или вызывать `$1c-workflow`.

## 12. Что делать при ошибках

Если агент сообщает, что Git-дерево не чистое, сначала решите, что делать с текущими изменениями: закоммитить, отложить или отменить.

Если не найден `1cv8.exe`, попросите агента показать установленные версии платформы 1С или заполните `PLATFORM_PATH` в `.dev.env` вручную.

Если нужен нестандартный каталог копий баз подпроектов, заполните `FEATURE_INFOBASE_ROOT` в `.dev.env` или `featureInfoBaseRoot` в `.agent-1c/project.json`. В обычном случае это не требуется.

Если агент сообщает, что файловая база не найдена или нет `1Cv8.1CD`, исправьте путь к базе. Не нажимайте "Да" в окне 1С "создать новую базу" для этого workflow.

Если не хватает Git, 1С platform или Apache/webinst, выполните предложенные агентом команды установки или настройте софт вручную.

Если 1С-команда завершилась ошибкой, смотрите путь к логу, который сообщил агент. Следующий шаг процесса нельзя выполнять, пока ошибка не исправлена.

Если возник конфликт при merge `master` в фичу, разрешите конфликт в файлах конфигурации, затем загрузите фичу в базу подпроекта и повторите проверку.

## 13. Правила безопасности

- Не храните пароли в Git.
- Не коммитьте `.dev.env`.
- Не загружайте изменения фичи напрямую в исходную базу, подключенную к хранилищу.
- Не удаляйте вручную каталог выгрузки конфигурации вне проекта.
- Не запускайте завершение фичи, пока она не протестирована.
- Перед переключением веток держите Git-дерево чистым.
