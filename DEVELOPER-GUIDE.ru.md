# Инструкция разработчика

> Документ для разработчика. Агентам не нужно читать его как обязательный lifecycle reference: канонические агентские правила находятся в `.agents/skills/1c-workflow/references/workflow.md`, а процесс работы внутри ветки - в `.agents/skills/1c-workflow/references/dev-branch-development.md`. Читайте этот файл только по просьбе пользователя или для подготовки human-facing объяснений.

Эта инструкция описывает стандартный процесс доработки 1С-конфигурации с помощью агента.

## 1. Основная идея

В проекте есть две сущности:

- `master` - локальная Git-ветка со свежей выгрузкой конфигурации из исходной базы. Исходная база может быть подключена к хранилищу 1С или обновляться вручную.
- `itldev/<name>` - ветка разработки для изолированной линии работы. По умолчанию у нее есть отдельная Git worktree-папка и собственная копия базы 1С, отвязанная от хранилища только если исходная база была подключена к нему.

Ветка разработки может использоваться не только под одну бизнес-фичу. Допустим сценарий, когда в одной ветке последовательно выполняется несколько задач, а изменения периодически передаются в исходную базу через выгруженный `CF`/`CFE`.

Для одновременной работы по нескольким независимым линиям создавайте отдельные `itldev/*` ветки/worktree. Одна ветка при этом может оставаться долгоживущей и последовательно вести несколько задач.

Агент никогда не загружает изменения из ветки разработки напрямую в исходную базу. Для переноса изменений агент готовит `CF` или `CFE`, а загрузка в исходную базу выполняется вручную.

## 2. Инициализация проекта

Откройте пустую или существующую папку проекта в Codex или Kilo Code и напишите:

```text
Инициализируй 1С-проект по файлу https://raw.githubusercontent.com/xmentosx/1c-agent-workflow/master/AGENT-INSTALL.md
```

Текущая папка будет папкой проекта. Основной путь инициализации - PowerShell script wizard: он покажет абсолютный путь, спросит подтверждение, соберет параметры и запустит весь lifecycle.

Во время инициализации script wizard спросит:

- версию платформы 1С, если ее нельзя выбрать из найденных установок;
- тип исходной базы: файловая или серверная;
- параметры исходной базы и, если она подключена к хранилищу, параметры хранилища;
- нужно ли публиковать базы веток разработки на Apache для веб-клиента;
- установить ли Vanessa Automation для исполняемых тестов веток разработки;
- настроить vibecoding1c MCP сразу или сделать это позже через `/itl-vibecoding1c-mcp`.

Агент отдельно спросит, подключена ли исходная база к хранилищу конфигурации. Если не подключена, вопросы про адрес, пользователя и пароль хранилища не задаются.

Для файловой базы, подключенной к хранилищу, параметры базы и хранилища задаются группой отдельных вопросов:

1. Путь к каталогу базы.
2. Пользователь базы.
3. Пароль базы или `нет`/`-`.
4. Адрес хранилища.
5. Пользователь хранилища.
6. Пароль хранилища или `нет`/`-`.

Для файловой базы без хранилища задаются только 3 вопроса: путь к каталогу базы, пользователь базы, пароль базы или `нет`/`-`.

Для серверной базы, подключенной к хранилищу:

1. Сервер 1С.
2. Имя базы.
3. Пользователь базы.
4. Пароль базы или `нет`/`-`.
5. Адрес хранилища.
6. Пользователь хранилища.
7. Пароль хранилища или `нет`/`-`.

Значения `нет` и `-` в поле пароля означают пустой пароль.

Для серверной базы без хранилища задаются только 4 вопроса: сервер 1С, имя базы, пользователь базы, пароль базы или `нет`/`-`.

После инициализации в `master` должен быть закоммичен каталог `src/cf` с файлом `src/cf/ConfigDumpInfo.xml`.

## 3. Где что хранится

- `src/cf` - выгрузка конфигурации в файлы, хранится в Git.
- `src/cfe` - выгрузки расширений в файлы, хранятся в Git.
- `.agent-1c/project.json` - настройки проекта без секретов.
- `.agent-1c/tools.json` - проверяемые инструменты и подсказки по установке, хранится в Git.
- `.agent-1c/dependency-lock.json` - lock manifest для ITL workflow-пакета, `ai_rules_1c`, Vanessa Automation и скачиваемых архивов; хранится в Git.
- `.dev.env` - локальные настройки и секреты, не коммитится.
- `.agent-1c/dev-branches/*.json` - локальное состояние веток разработки, не коммитится.
- `<папка-проекта>-worktrees/<ветка>` - отдельные Git worktree для параллельных линий разработки, создаются рядом с основной папкой проекта.
- `.agent-1c/infobases/dev-branches` внутри worktree - копии файловых баз веток разработки, не коммитятся.
- `build/result` - выгруженные `CF`/`CFE` и соседние `<artifact>.manifest.json`, не коммитятся.
- `tests/features` - Vanessa Automation сценарии, хранятся в Git.
- `build/test-results/vanessa` - отчеты Vanessa Automation, не коммитятся.
- `.agent-1c/tools/vanessa-mcp` - скачанные `client_mcp.cfe` и `VAExtension.*.cfe`, не коммитятся.
- `.agent-1c/mcp/` - локальное состояние MCP текущего проекта/worktree и выбор remote/local, не коммитится.
- `.codex/config.toml`, `.kilo/kilo.json`, `.kilo/kilo.jsonc` - локальные MCP client config, не коммитятся.
- `VANESSA_TEST_PORT` в `.dev.env` - branch-local порт `TESTMANAGER -> TESTCLIENT` для `/itl-verify`, управляется helper.
- `VANESSA_MCP_PORT` и `VANESSA_MCP_URL` в `.dev.env` - активная MCP-точка текущей ветки, управляется helper.
- `logs/1c` - логи запусков 1С, не коммитятся.

## 4. Создание ветки разработки

В Kilo Code:

```text
/itl-new-config-branch СкидкиПоЗаказам
```

Или обычным текстом:

```text
Создай ветку разработки "Скидки по заказам"
```

Агент должен:

1. Проверить, что Git-дерево чистое.
2. Убедиться, что основная папка проекта находится на `master`.
3. Создать ветку `itldev/<name>` в отдельной sibling worktree-папке.
4. Скопировать `.dev.env` в worktree.
5. Скопировать исходную базу в `.agent-1c/infobases/dev-branches/<name>` внутри worktree.
6. Отвязать копию базы от хранилища, если исходная база была подключена к хранилищу.
7. Добавить копию базы в список баз 1С в папку `/ITL/<имя-папки-проекта>` с именем записи, равным имени ветки разработки.
8. При включенной Apache-публикации опубликовать базу.
9. Сохранить локальное состояние ветки в `.agent-1c/dev-branches/<name>.json` внутри worktree.
10. Сообщить путь worktree и предложить открыть отдельное окно агента в этой папке.

Основная папка проекта после создания остается на `master`. Все дальнейшие изменения загружаются только в базу этой ветки разработки и выполняются из worktree-папки этой линии работы.

### Ветка разработки расширения

Если в ветке нужно разрабатывать расширение, создайте отдельную extension-ветку:

```text
/itl-new-extension-branch МоеРасширение
```

При создании ветки имя расширения не указывается и само расширение в базе не создается. В начале разработки задайте имя расширения:

```text
/itl-set-dev-branch-extension МоеРасширение
```

Затем создайте расширение в копии базы этой ветки через Конфигуратор. После создания выгрузите его в файлы:

```text
/itl-dump-dev-branch-extension
```

Файлы расширения появятся в `src/cfe/<имя-расширения>`. Дальше изменения в файлах расширения обновляются в базе ветки обычной командой `/itl-update-base`, а итоговый `CFE` выгружается через `/itl-result`.

`/itl-set-dev-branch-extension` и `/itl-dump-dev-branch-extension` - helper-команды для extension-веток. Они доступны напрямую, но не входят в короткое beginner-меню `/itl`.

## 5. Работа внутри ветки

Процесс разработки внутри ветки описан отдельно:

```text
DEV-BRANCH-DEVELOPMENT.ru.md
```

Короткое правило:

- маленькое локальное исправление можно делать как quick-fix;
- более широкие, рискованные или неоднозначные изменения оформляются через OpenSpec.

После изменения файлов конфигурации обновите базу ветки:

```text
/itl-update-base
```

Или:

```text
Обнови базу текущей ветки разработки
```

Агент обновляет базу ветки только измененными файлами из `src/cf`, формируя `listFile` и передавая его в 1С через `/LoadConfigFromFiles ... -listFile ... -Format Hierarchical`.

Для стандартной проверки обновите базу и запустите Vanessa Automation одной командой:

```text
/itl-verify
```

Для написания и отладки Vanessa-сценариев можно использовать branch-local Vanessa MCP через `/itl-vanessa-mcp` из worktree нужной `itldev/*` ветки. MCP помогает исследовать формы, подбирать шаги и отлаживать сценарий, но финальная проверка ветки остается через `/itl-verify`: helper запускает пакетный `StartFeaturePlayer` в реальном `TESTMANAGER -> TESTCLIENT` контуре, назначает отдельный test port текущей ветки и проверяет JUnit-отчет.

vibecoding1c MCP для поиска по документации, шаблонам, синтаксису, коду и графу подключается через `/itl-vibecoding1c-mcp`. По умолчанию setup применяет сохраненный выбор серверов, а если выбор отсутствует или неполный - сначала запускает выбор; для явного повторного выбора используйте `vibecoding1c-mcp-select` или `vibecoding1c-mcp-setup -Force`. Затем helper берет remote LAN endpoints из GitLab registry `http://gitlabserv01.itland.local/root/MCP-vibecoding1c-registry.git`; для серверов по конкретной конфигурации нужно явно выбрать `configId`, даже если сейчас опубликована одна конфигурация. По каждому серверу можно переключиться на local. Для local `code`/`graph` выбирается scope `project` или `branch`. Vanessa MCP относится к отдельному семейству, запускается только через `/itl-vanessa-mcp` и всегда branch-local. `/itl-status` показывает freshness vibecoding1c MCP: `fresh`, `stale`, `remote-shared`, `unknown` или `indexing`, поэтому агент видит, когда индекс не отражает текущие изменения ветки.

Для изменения бизнес-поведения агент создает или обновляет небольшой набор Vanessa Automation проверок: минимум 2, обычно 2-3 и не больше 4 без отдельного обоснования. Набор должен включать основной успешный сценарий и минимум один значимый граничный или негативный сценарий. Тип проверки выбирается по сути изменения: unit-like для локальной логики, integration для взаимодействия объектов/регистров/документов/обменов, UI только для форм, команд и видимого пользовательского поведения.

Не используйте `/deploy-and-test` как обычный шаг проверки в ITL-ветке: он повторно загружает всю конфигурацию. Если нужно только обновить базу без тестов, используйте `/itl-update-base`.

## 6. Обновление ветки свежим master

Если в хранилище или в исходной базе появились изменения от других разработчиков:

```text
/itl-refresh
```

Агент должен:

1. Проверить чистоту Git-дерева.
2. Если исходная база подключена к хранилищу, обновить ее из хранилища. Если не подключена, пропустить этот шаг: перед командой разработчик должен сам перенести свежие изменения в исходную базу.
3. Выгрузить свежий `src/cf` в `master`.
4. Закоммитить изменения `master`, если они есть.
5. Вернуться в текущую ветку разработки.
6. Смержить `master` в ветку разработки.
7. Обновить базу ветки измененными файлами.

Если возникнут конфликты, агент должен остановиться и помочь их разрешить.

## 7. Выгрузка результата без закрытия ветки

Когда нужно передать текущую версию изменений в исходную базу:

```text
/itl-result
```

Агент:

1. Проверит чистоту Git-дерева.
2. Убедится, что работает с текущей веткой разработки.
3. Обновит базу ветки измененными файлами.
4. Выгрузит `CF` или `CFE` в `build/result`.
5. Создаст рядом `<artifact>.manifest.json` с SHA256, commit-ами, статусом проверки и логами.
6. Сообщит путь к файлу и manifest.

Эта команда не обновляет `master` и не закрывает ветку.

Если нет свежей успешной проверки Vanessa, helper применит `VERIFICATION_POLICY`. В режиме `warn` агент покажет предупреждение, и продолжить выгрузку результата можно только после явного подтверждения разработчика. В режиме `block` результат нельзя выгрузить до fresh passed `/itl-verify`.

## 8. Список и переход между ветками

Показать активные ветки разработки:

```text
/itl-status
```

Перейти к `master` или ветке разработки:

```text
/itl-switch master
/itl-switch СкидкиПоЗаказам
```

Для worktree-веток `/itl-switch` не перезаписывает текущую папку через `git checkout`. Команда показывает путь нужной рабочей папки и предлагает открыть ее отдельным окном агента. Старое checkout-поведение используется только для legacy-веток, созданных с `-UseCurrentWorktree`.

## 9. Закрытие ветки разработки

Когда работа в ветке завершена и протестирована:

```text
/itl-close
```

Агент должен:

1. Проверить чистоту Git-дерева.
2. Обновить `master` из хранилища или из текущего состояния исходной базы.
3. Смержить `master` в ветку разработки.
4. Обновить базу ветки итоговыми изменениями.
5. Выгрузить финальный `CF` или `CFE` в `build/result`.
6. Создать рядом `<artifact>.manifest.json`.
7. Записать `closedAt` в локальное состояние ветки.
8. Сообщить путь к `CF`/`CFE`, manifest, commit ветки, commit `master` и путь к последнему логу 1С.
9. Оставить основную папку проекта на `master`; worktree, база и локальный state не удаляются автоматически.

Если нет свежей успешной проверки Vanessa, helper применит `VERIFICATION_POLICY`. В режиме `warn` агент покажет предупреждение, и закрыть ветку можно только после явного подтверждения разработчика. В режиме `block` закрытие запрещено до fresh passed `/itl-verify`.

## 10. Полезные команды

```text
/itl                              Показать меню действий.
/itl-new-config-branch <name>     Создать worktree-ветку разработки конфигурации.
/itl-new-extension-branch <name>  Создать worktree-ветку разработки расширения.
/itl-vibecoding1c-mcp            Настроить, запустить, обновить или проверить vibecoding1c MCP.
/itl-status                       Показать текущую ветку, базу и статус проверки.
/itl-update-base                  Обновить базу текущей ветки из файлов Git-ветки.
/itl-verify                       Обновить базу ветки и запустить Vanessa Automation.
/itl-refresh                      Обновить ветку свежим master.
/itl-result                       Выгрузить CF/CFE без закрытия.
/itl-close                        Закрыть текущую ветку и выгрузить результат.
/itl-switch <master|branch name>  Показать/открыть нужную рабочую папку или переключить legacy-ветку.
```

## 11. Правила безопасности

- Не коммитьте `.dev.env`, пароли, `*.cf`, `*.dt`, логи и локальные базы.
- Не коммитьте `.agent-1c/dev-branches/*.json`: это локальное runtime-состояние веток.
- Не загружайте изменения ветки разработки напрямую в исходную базу.
- Не используйте один Vanessa MCP на несколько веток разработки: у каждой `itldev/*` worktree должен быть свой сохраненный MCP порт и URL. MCP не является финальным verify-runner.
- Перед командами `ai_rules_1c`, которые работают с базой (`/update1cbase`, `/loadfrom1cbase`, `/getconfigfiles`), в ветке `itldev/*` должен быть активирован контекст ветки. Команды жизненного цикла делают это автоматически.
- `/deploy-and-test` в ITL-ветке запускайте только по явной просьбе разработчика, потому что он делает полную загрузку файлов. Для обычной проверки используйте `/itl-verify`; чужие Vanessa `TESTMANAGER`/`TESTCLIENT` процессы в другой worktree по умолчанию только показываются как диагностика и не блокируют запуск, если нет конфликта порта или базы. Helper не завершает процессы другой worktree.
- `/itl-result` и `/itl-close` следуют `VERIFICATION_POLICY`: по умолчанию `warn` предупреждает при отсутствии свежей успешной проверки Vanessa и требует явное подтверждение; `block` запрещает продолжение до `/itl-verify`.
- Перед созданием worktree, обновлением ветки, обновлением `master`, выгрузкой результата и legacy-переключением Git-дерево должно быть чистым.
- Если 1С или Git вернули ошибку, агент должен остановиться и показать путь к логу.
- По умолчанию `DEPENDENCY_MODE=fresh`: workflow берет свежие зависимости и обновляет `.agent-1c/dependency-lock.json`. Для воспроизводимого bootstrap используйте `DEPENDENCY_MODE=locked` с заполненными pin/hash значениями.
- Чтобы обновить сам установленный ITL workflow-пакет в уже развернутом проекте, запускайте `/itl-update-workflow` или helper-действие `update-workflow` из `master` worktree. Команда обновляет только управляемые workflow-файлы, не трогает локальное состояние (`.dev.env`, `.agent-1c/dev-branches/`, `.agent-1c/mcp/`, `.codex/config.toml`, `.kilo/kilo.json*`, существующие `.agent-1c/project.json` и `.agent-1c/tools.json`), по умолчанию запускает `update-ai-rules` и оставляет изменения для review/commit. Уже созданные `itldev/*` ветки обновляйте отдельно через merge свежего `master` или `/itl-refresh`; runtime MCP обновляйте через `/itl-vibecoding1c-mcp`, а branch-local Vanessa MCP - через stop/install/start в нужной worktree.
- Чтобы получить свежую версию upstream-правил `ai_rules_1c`, используйте `/itl-update-rules` или helper-действие `update-ai-rules`. Helper убирает дефолтные upstream MCP-записи из локальных Codex/Kilo конфигов, ITL-надстройка повторно применяется в `USER-RULES.md`, а root `AGENTS.md` остается управляемым upstream-инсталлером, если он уже ссылается на `USER-RULES.md`.
